The `<SegmentedControl>` lets people pick one option from a small set of choices that are all visible at once. It is a compact, horizontal control built for switching a view, a mode, a sort order, or a filter, usually with an immediate effect.

It is also a proper form field. It has a label, can show a description and an error message, and can be marked as required or read only, just like a [Radio.Group](/components/form/radio).

Reach for a [Select](/components/form/select) when there are more than a handful of options or space is tight. Reach for [Tabs](/components/navigation/tabs) when you are moving between sections of content. Reach for a [ToggleButton.Group](/components/actions/toggle-button) when the buttons are independent on and off actions rather than one exclusive choice.

## Anatomy

A `<SegmentedControl>` is a labeled track that holds two or more segments in a row. Only one segment is selected at a time, marked by an indicator that moves to the chosen segment.

<SegmentedControlAnatomy />

- **Label:** Names the control and explains what the choice affects.
- **Track:** The container that groups the segments into one unit.
- **Segment:** A single option, added with `<SegmentedControl.Option>`. It holds a short text label, an icon, or both.
- **Selection indicator:** The highlight that marks the selected segment and moves as the selection changes.
- **Description:** Optional helper text below the track. It is replaced by the error message when something is wrong.

The control comes in two variants and two sizes so it fits both prominent and dense places.

| Variant   | Description                                                                                         | When to use                                                                    |
| --------- | --------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `default` | A filled track with a raised indicator, matching the look of the [Switch](/components/form/switch). | Standalone controls above a list or table, or inside a form or card.           |
| `ghost`   | A style without a filled track and a translucent indicator that blends into the surface behind it.  | Dense toolbars and panel headers where the control sits next to other actions. |

| Size      | Description                                  | When to use                           |
| --------- | -------------------------------------------- | ------------------------------------- |
| `default` | Standard height, matching the input height.  | General purpose use.                  |
| `small`   | Reduced height and text for compact layouts. | Toolbars and space constrained areas. |

## Usage

A segmented control shows a single current choice among a few options. Selecting a segment usually takes effect right away. It changes how content is shown, sorted, or filtered, instead of waiting for a form to be sent. Always start with one option already selected, so the control is never empty.

### Switching views or modes

The most common use is letting people switch how the same content is shown, for example between a list, a board, and a calendar view. Keep the control directly above the content it affects so the connection is clear.

<ComponentDemo name="segmentedcontrol-basic" />

### Number of options

A segmented control works best with **two to five** options. Beyond that, the labels get cramped, the segments become hard to tap, and the control is likely to overflow its container. When you have more options, switch to a [Select](/components/form/select) to save space, or to [Tabs](/components/navigation/tabs) when the choice is really about moving between sections of content.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use a segmented control for two to five options that are short and can all
      be read at a glance.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't crowd a segmented control with many options. Use a
      [Select](/components/form/select) when the list grows beyond a handful.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Labels

Use short, parallel labels, ideally one or two words, so every segment reads consistently and the control stays compact. Don't shorten a label with an ellipsis. If a label is too long, reword it or pick a different component. Keep the option order stable and logical, following a natural sequence such as Day, Week, Month, or ordering by importance, and never reorder options based on the current selection.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Keep labels short and parallel, and keep their order stable.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't use long labels or shorten them with an ellipsis. Reword the label
      or use a [Radio.Group](/components/form/radio) instead.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Full width

By default the control is only as wide as its segments. Set `width="full"` to make the segments share the available width equally. This is useful inside a card, a sidebar, or a narrow panel where the control should fill its container.

<ComponentDemo name="segmentedcontrol-full-width" />

### Toolbars

In dense places like a toolbar or a panel header, use the `ghost` variant, often together with `size="small"`. It drops the filled track so the control blends in with the surrounding actions while still marking the current selection.

<ComponentDemo name="segmentedcontrol-ghost" />

### Overflow on small screens

When the segments don't fit, usually on narrow screens, the control scrolls sideways instead of squeezing the segments. It keeps the selected option in view and fades its edges to show there is more to scroll. Treat this as a graceful fallback rather than a reason to add options. If a control often overflows, it has too many options for a segmented control.

<ComponentDemo name="segmentedcontrol-overflow" />

<Callout title="Keep it accessible">
  `<SegmentedControl>` works like a group of radio buttons. People move between
  segments with the arrow keys, and the selection follows focus. Tab moves into
  and out of the control as a single stop. Always provide a `label`, or an
  `aria-label` in a toolbar, so the purpose of the control is announced.
</Callout>

### Choosing the right component

A segmented control overlaps with several other components because it lets people make a single choice. The table below sums up when to reach for it and when another component fits better.

| If you need to…                                                               | Use                                                     |
| ----------------------------------------------------------------------------- | ------------------------------------------------------- |
| Switch between a few short views, modes, or filters that are all visible      | **SegmentedControl**                                    |
| Collect one choice in a form, with longer labels or more than about 5 options | [Radio.Group](/components/form/radio)                   |
| Choose one option from a moderate or long list, or save vertical space        | [Select](/components/form/select)                       |
| Move between sections of content within the same view                         | [Tabs](/components/navigation/tabs)                     |
| Toggle several independent on and off actions in a toolbar                    | [ToggleButton.Group](/components/actions/toggle-button) |
| Turn a single setting on or off                                               | [Switch](/components/form/switch)                       |

<GuidelineTiles>

      Use a segmented control for a single, mutually exclusive choice among a
      few options that should all stay visible, like switching a view or a
      filter.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't use it for independent on/off actions where more than one can be

      instead.
    </DontDescription>
  </Dont>
  <Dont>
    <DontDescription>

  </Dont>

    <DontDescription>
      Don't use a two-segment control to represent a simple on/off state. Use a

    </DontDescription>

</GuidelineTiles>

## Alternative components

- [Radio.Group](/components/form/radio): The same single choice in a traditional, vertical form layout. Use it when labels are long or need descriptions, when there are more than a handful of options, or when the choice is sent with the rest of the form rather than taking effect right away.
- [Select](/components/form/select): Hides the options behind a dropdown. Use it for a moderate number of options, around 5 to 15, or when space is tight.
- [Tabs](/components/navigation/tabs): Moves between sections of content that live in the same place and shows the matching panel. Use it when the choice is about navigation, not setting a value.
- [ToggleButton.Group](/components/actions/toggle-button): A toolbar of independent on and off actions where any number can be active. Use it for several choices at once rather than one exclusive choice.
- [Switch](/components/form/switch): Turns a single setting on or off with an immediate effect.
