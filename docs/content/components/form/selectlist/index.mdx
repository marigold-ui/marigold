A `<SelectList>` lets users pick from a list where each option carries additional information beyond a label, like a description, a logo, or a per-row action. Every option stays visible so users can scan and compare without opening anything.

The trade-off is space. A `<SelectList>` takes more vertical room than a `<Radio.Group>` or a `<Select>`, so reach for it when the decision earns the weight: picking a payment method, comparing plans, choosing a shipping speed. For simple text choices, a lighter control reads faster.

## Anatomy

A `<SelectList>` is a labeled container holding a stack of `<SelectList.Option>` rows. Each row pairs a selection indicator with a label over an optional description, and an optional trailing action.

<SelectListAnatomy />

- **Label:** Names the field above the list.
- **List:** The container that holds all options.
- **Option:** A single row representing one choice.
- **Selection indicator:** A radio circle for single selection, or a checkbox for multiple selection.
- **Text value:** The primary text of an option.
- **Description:** Optional secondary text within an option that helps the user decide.
- **Action:** An optional trailing button or menu for a per-row action that does not toggle the row.

`<SelectList>` has two variants. They handle two different framings of the same choice: rows in a continuous list, or peer cards to commit to.

| Variant    | Description                                         | When to use                                                                        |
| ---------- | --------------------------------------------------- | ---------------------------------------------------------------------------------- |
| `default`  | A single surface with dividers between rows.        | Picking from a defined set inside a form. Reads as one continuous list.            |
| `bordered` | Each row is its own outlined card with a small gap. | Commit-level, peer-to-peer choices like plans, payment methods, or shipping modes. |

## Usage

`<SelectList>` sits between a `<Radio.Group>` and a `<Table>`: more context per option than a radio, less structural weight than a table. Use the option count and the kind of choice to decide which control fits.

- **2–4 plain text choices:** prefer `<Radio.Group>` (single) or `<Checkbox.Group>` (multiple). Lighter and faster to scan.
- **5–10 choices where each option needs context** (a description, a logo, or a trailing action): `<SelectList>` earns its space.
- **More than ~15 options, or users need to search:** prefer [`<Select>`](/components/form/select) (single), [`<Multiselect>`](/components/form/multiselect), or [`<TagField>`](/components/form/tag-field).
- **Records with multiple structured columns:** use a [`<Table>`](/components/collection/table). `<SelectList>` is for picking, not for displaying data.

### Single vs multiple selection

A `<SelectList>` defaults to single selection. Each row renders a radio circle, and selecting one row replaces the previous selection. Like a `<Radio.Group>`, the user cannot clear the selection by clicking the active row again.

When several options can be active at once, set `selectionMode="multiple"`. Rows render checkboxes and toggle independently. This is also the natural moment for per-row actions: drop an `<IconButton>` or `<ActionMenu>` into a `<SelectList.Option>` and it sits on the trailing edge of the row. Keep it to one action per row so the row's tap target stays predictable. Clicking the action does not toggle the row.

If a row needs more than one action, group them inside a single `<ActionMenu>` instead of placing two icons side by side.

### Bordered variant

A flat list with dividers reads as "pick one of these entries"; bordered cards read as "commit to one of these". When each option represents a commit-level choice, like a plan, a payment method, or a shipping mode, use `variant="bordered"`. Each row becomes its own outlined card with a small gap between cards. The selected card strengthens its border instead of filling, which suits options that should feel like peer choices rather than entries in a continuous list.

### Clearable single selection

By default, `selectionMode="single"` does not let users clear the selection by clicking the active row, matching radio-group behavior. Some scenarios benefit from the opposite: filter pickers where "no filter" is a valid state, or optional pickers where the user must be able to back out of a choice. In those cases, set `disallowEmptySelection={false}` so clicking the active row clears it.

<ComponentDemo name="selectlist-clearable" />

### Horizontal orientation

For short lists where comparing options side by side makes sense, set `orientation="horizontal"`. The list flows left to right and the surface shrinks to fit the content. Arrow key navigation follows the orientation: <kbd>Left</kbd> and <kbd>Right</kbd> move between options instead of <kbd>Up</kbd> and <kbd>Down</kbd>.

Horizontal works best for compact choices like shipping speed or sort order. Avoid it on small viewports, where horizontal scrolling is hard to discover.

<ComponentDemo name="selectlist-horizontal" />

### Empty state

When the collection is empty, for example because the user hasn't saved any payment methods yet, render a placeholder via the `emptyState` prop. A good empty state explains why the list is empty and offers a single next step.

When the list is empty because of an active filter or search, surface a "Clear filters" action so users can recover without leaving the field.

### Working in forms

`<SelectList>` is a form field. Set the `name` prop and the component participates in standard form submission. Single selection submits one value; multiple selection submits all selected keys under the same name, so `FormData.getAll(name)` returns the full array.

Use `required` to mark the field as required, and `validate` for custom validation logic. When the form is submitted with native validation, errors focus the list automatically.

<ComponentDemo name="selectlist-form" />

### Disabling options

Two layers of disabling work together. Pass `disabled` on the `<SelectList>` to lock the entire field, for example while another step is still loading. Pass `disabled` on a single `<SelectList.Option>` to remove that one row from selection.

By default, a disabled row is also unfocusable. Set `disabledBehavior="selection"` when you still want users to focus the row to reach its trailing action, for example to open an `<ActionMenu>` and remove a card that cannot currently be selected.

### Writing good options

Keep option labels short and concrete so users can scan the list quickly. Lead with the most distinctive word, since that's where the eye lands first. The description line earns its space when it differentiates options that look similar at the title level. Skip it for marketing copy, restated labels, or boilerplate that repeats on every row ("All major cards accepted").

Be consistent. All rows should follow the same shape: either every row has a description or none of them do. Mixed-height rows break the visual rhythm and slow scanning.

<Callout title="Keep it accessible">
  When a row contains more than plain text (a wrapped label, a logo, a trailing
  action), pass `textValue` on the option. Screen readers use it as the option's
  accessible name.
</Callout>

<GuidelineTiles>
  <Do>
    <DoDescription>
      Keep labels short and lead with the most distinctive word. Use the
      description line for information that helps the user decide.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't mix one-line and two-line rows in the same list, and don't truncate
      labels with an ellipsis.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Ordering options

Put the option users pick most often first, since they scan top to bottom (or left to right). Avoid alphabetical or system-driven order: a "Bank transfer" sorted to the top alphabetically is unhelpful when most users pay by card. If options have a natural progression (free → pro → enterprise, or fastest → cheapest), follow that order instead.

## Accessibility

`<SelectList>` is fully keyboard accessible. Arrow keys move focus between rows, <kbd>Space</kbd> toggles the focused row, and <kbd>Tab</kbd> reaches any action inside it. This two-tier model is what lets a per-row button or menu coexist with row selection without breaking keyboard navigation.

Selection state is announced semantically, never through color alone. When `selectionMode="multiple"`, the list is announced as multi-selectable, and disabled rows and disabled fields are announced as disabled.

For rows with rich content, pass `textValue` on the option so screen readers announce the right name. Without it, the announcement collapses to whatever text fragment is reachable, which is rarely correct.

## Alternative components

- [Radio](/components/form/radio): Use for a small set of mutually exclusive, text-only choices where a description per option is not needed.
- [Checkbox](/components/form/checkbox): Use for a small set of independent boolean options where each option is a short label.
- [Select](/components/form/select): Use when the list is long, options are single-line, and saving vertical space matters more than seeing every option at once.
- [TagField](/components/form/tag-field): Use for multi-select with search, where users add and remove choices as tags.
- [Table](/components/collection/table): Use when each row is a record with multiple columns of structured data, not a selectable choice.
