A `<ListView>` renders a non-form list of rows the user acts on without leaving the page: mute a notification, dismiss an alert, rename a resource, flip a setting. Unlike [`<SelectList>`](/components/form/selectlist) or [`<Select>`](/components/form/select), a `<ListView>` never becomes a submitted value — there is no selection, no hidden input, no `name`/`form`/`validate`. It exists purely so rows can host interactive controls (a `<Switch>`, an `<IconButton>`, an `<ActionMenu>`, a link) with correct grid semantics and keyboard navigation.

## Anatomy

A `<ListView>` is a labeled, non-form grid holding a stack of `<ListView.Item>` rows. Each row is authored as flat children: an optional leading icon or image, the row's text (`<TextValue>`/`<Description>`, or `<Title>` for a heading-like row), and optional trailing controls.

- **List:** The grid container. Requires `aria-label` or `aria-labelledby`.
- **Item:** A single row (`role="row"`). Not selectable or toggleable by ListView itself.
- **Leading content:** An optional icon or image before the row's text.
- **Text value:** The row's primary text, authored with `<TextValue>` (or `<Title>` for a heading-like row — see [Accessibility](#accessibility)).
- **Description:** Optional secondary text below the text value.
- **Trailing controls:** One or more interactive controls — `<Switch>`, `<IconButton>`, `<ActionMenu>`, a link — positioned at the end of the row.

## Usage

### Which list component?

Marigold has several list-shaped components. Picking the wrong one either strips accessibility semantics your rows need, or adds semantics they don't. Match the component to what the user does with a row:

| The user is going to…                                        | Component                                                                             |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------------- |
| Read a static list                                           | [`<List>`](/components/content/list)                                                  |
| Pick one/some from a set (it becomes a value)                | [`<SelectList>`](/components/form/selectlist) / [`<Select>`](/components/form/select) |
| Fire a command that closes the surface                       | [`<Menu>`](/components/actions/menu)                                                  |
| Compare rows across columns                                  | [`<Table>`](/components/collection/table)                                             |
| Operate items in place (toggle / dismiss / act), staying put | `<ListView>`                                                                          |

`<ListView>`'s closest neighbor is a non-form selection list built on `ListBox`/`option` semantics: `listbox`/`option` forbids focusable controls inside a row, since the row itself is the only target (click it to pick). `<ListView>` uses `grid`/`row`/`gridcell` semantics instead, which is required once rows hold controls. They are not redundant — `listbox`/`option` is the lighter, correct semantic for pure selection (a grid announcing every pick-one row is needlessly verbose for screen readers), while `grid`/`row` is what interactive rows require.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Reach for `<ListView>` when a row needs its own button, switch, or menu
      that the user operates without navigating away.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't use `<ListView>` when clicking the row itself is the only
      interaction — that's a selection list (`<ListBox>` or `<SelectList>`),
      not `<ListView>`.
    </DontDescription>
  </Dont>
</GuidelineTiles>

For a very short list (roughly 5 rows or fewer) where each row has at most one control, a plain `<Stack>` of focusable rows is lighter and perfectly fine. `<ListView>` earns its keep on longer lists, or rows with multiple controls, where roving keyboard navigation and consistent list accessibility actually pay off.

## Accessibility

`<ListView>` is fully keyboard accessible:

- Arrow keys move focus between rows.
- Tab reaches any interactive control inside the focused row.
- Rows are never selectable or toggleable by `<ListView>` itself — operating a nested control is the only interaction a row exposes.

Always give the list an `aria-label`, or pass `aria-labelledby` when a heading nearby already names it.

For rows with rich content — a wrapped label, an icon, a trailing control — set `textValue` on the item. Screen readers use it as the row's accessible name.

<Callout title="Author row text with the dedicated slot components">
  Use `<TextValue>` and `<Description>`, never a raw `<Text slot="…">`.
  `<ListView.Item>` provides the matching slot contexts internally, so these
  thin wrappers resolve correctly without any extra wiring.
</Callout>

`<Title>` renders a real `<h1>`–`<h6>`. A list must not emit one document heading per row — that destroys the heading outline for screen reader users navigating by heading. `<ListView.Item>` forces every nested `<Title>` to render as a `<span>` instead, so it's safe to reach for when a row is genuinely heading-like; the default row text remains `<TextValue>`.

## Alternative components

- [List](/components/content/list): Use for a static, presentational list with no roles, keyboard model, or interactive rows.
- ListBox: Marigold's non-form selection list (used internally by [Combobox](/components/form/combobox) and [Select](/components/form/select)). Use when clicking a row is the only interaction — pure selection, not operating controls in place.
- [SelectList](/components/form/selectlist): Use when the selection becomes a submitted form value.
- [Select](/components/form/select): Use for a compact single/multi-select form field.
- [Table](/components/collection/table): Use when each row is a record with multiple columns of structured data to compare, not a flat list of operable items.
- [Menu](/components/actions/menu): Use for a set of commands that close the surface on activation.
