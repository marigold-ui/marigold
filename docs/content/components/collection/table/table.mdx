The Table component displays structured data in a grid layout with rows and columns. It provides features for selecting, sorting, editing, and reordering data, making it ideal for applications that need to present and manipulate tabular information.

Use tables when you need to display datasets where users benefit from seeing multiple records at once, comparing values across columns, or performing bulk operations on selected items. Tables work well for admin interfaces, data management screens, and dashboards where structured information needs to be organized and actionable.

## Anatomy

<Image

- **Table:** Main container that wraps all table elements.
- **Header (sticky):** Container for column headers, can stick to viewport top when scrolling.
- **Column with sorting:** Individual column headers that can enable sorting functionality.
- **Selection checkbox:** Checkbox for selecting individual rows or all rows at once.
- **Row (selectable):** Individual table rows that can be selected, clicked, or dragged.
- **Cell:** Standard cell containing data, text, or components.
- **Editable Cell (with edit icon):** Cell that can be edited inline with a pencil icon trigger.
- **Drag handle:** Visual indicator for drag and drop reordering of rows.

## Appearance

The Table component offers different visual styles to match your use case and data density.

- **Default:** Standard table appearance for most use cases with clean, clear data presentation.
- **Muted:** Reduces visual noise in tables with extensive content, helping users focus on the data.
- **Grid:** Adds vertical borders between columns for clearer separation, useful for tables with many columns or dense data.
- **Admin and Master:** Identifies rows relevant only for internal users. [Learn more about its usage.](/patterns/admin-master-mark)

## Usage

Tables organize information into a grid structure that makes data easy to scan, compare, and act upon. Use tables when you need to present multiple records with consistent attributes, allow users to select and perform actions on items, or enable sorting and filtering of large datasets.

The Table component supports various interaction patterns including row selection for bulk operations, column sorting for data organization, inline editing for quick updates, and drag and drop for manual reordering. Choose the features that match your use case without overwhelming users with unnecessary functionality.

### Sorting columns

When working with large datasets, users often need to organize information by specific attributes. Enable sorting to let users click column headers and reorder rows by that column's values.

<ComponentDemo file="./table-sorting.demo.tsx" />

Set `allowsSorting` on columns that should be sortable. Manage sort state with `sortDescriptor` and `onSortChange`. The component displays sort direction indicators (ascending/descending arrows) in column headers. Implement the sorting logic in your `onSortChange` handler, either client-side for small datasets or server-side for large ones.

<GuidelineTiles>
  <Do>
    <Do.Description>
      Enable sorting on columns with comparable data like numbers, dates, or
      alphabetical text.
    </Do.Description>
  </Do>
  <Do>
    <Do.Description>
      Right-align numeric columns to make values easier to compare when sorted.
    </Do.Description>
  </Do>
  <Dont>
    <Dont.Description>
      Don't make every column sortable if some contain non-comparable content
      like icons or actions.
    </Dont.Description>
  </Dont>
  <Dont>
    <Dont.Description>
      Don't forget to handle sorting server-side for large datasets to maintain
      performance.

### Editing data inline

When users need to make quick edits to individual values without navigating to a separate form, use editable cells. This reduces friction and keeps users in context while maintaining a clear editing workflow.

<ComponentDemo file="./table-editable.demo.tsx" />

Use `Table.EditableCell` instead of `Table.Cell` for editable fields. Provide a form control via `renderEditing` that appears when users click the edit icon. The editing interface adapts responsively, showing a Dialog on mobile and a Popover on desktop. Handle form submission with `onSubmit` to update your data.

<GuidelineTiles>
  <Do>
    <Do.Description>
      Use editable cells for quick, single-value updates like changing status or
      correcting typos.
    </Do.Description>
  </Do>
  <Do>
    <Do.Description>
      Provide appropriate input types like TextField for text, Select for
      constrained options.
    </Do.Description>
  </Do>
  <Dont>
    <Dont.Description>
      Don't make cells editable if editing requires complex validation or
      multiple related fields.

  </Dont>
  <Dont>
    <Dont.Description>
      Don't use editable cells when a dedicated edit form would provide better

### Reordering with drag and drop

When the order of items matters and users need to manually arrange them, enable drag and drop. This works well for priority lists, custom orderings, or workflows where sequence is important.

Use `useDragAndDrop` from react-aria-components and `useListData` from react-stately to manage drag operations. Pass `dragAndDropHooks` to the Table and implement `onReorder` to update your data. The component shows drag handles on hover and visual indicators for drop positions.

### Making headers sticky

For tables with many rows that require scrolling, sticky headers keep column labels visible as users scroll through data. This helps users maintain context about what each column represents.

<ComponentDemo file="./table-sticky.demo.tsx" />

Add the `sticky` prop to `Table.Header` and wrap the table in a `Scrollable` component with a defined height. Headers will stick to the top of the viewport as users scroll through rows, maintaining visibility of column labels.

<GuidelineTiles>
  <Do>
    <Do.Description>
      Use sticky headers for tables with more than 10-15 rows where scrolling is
      necessary.
    </Do.Description>
  </Do>
  <Do>

      behavior works correctly.
    </Do.Description>
  </Do>
  <Dont>

### Aligning content

Proper text alignment makes tabular data easier to scan and compare. Different data types benefit from different alignments to improve readability and understanding.

<ComponentDemo file="./table-alignment.demo.tsx" />

Use `align="left"` for text content (names, descriptions), `align="right"` for numeric values (prices, quantities, percentages), and `align="center"` sparingly for icons or status indicators. Apply alignment to both columns and cells for consistency.

<GuidelineTiles>
  <Do>
    <Do.Description>
      Right-align numeric columns to make decimal points and place values line
      up vertically.
    </Do.Description>
  </Do>
  <Do>
    <Do.Description>
      Left-align text content for natural reading flow in LTR languages.
    </Do.Description>
  </Do>
  <Dont>
    <Dont.Description>
      Don't center-align numbers or long text, as this makes scanning and
      comparison difficult.
    </Dont.Description>
  </Dont>
  <Dont>
    <Dont.Description>
      Don't mix alignments within a single column, as inconsistency breaks
      visual rhythm.
    </Dont.Description>
  </Dont>
</GuidelineTiles>

### Allowing text selection

By default, when selection mode is enabled, clicking on cell text selects the row. When users need to copy cell content like emails, IDs, or other data, enable text selection.

<ComponentDemo file="./table-text-selection.demo.tsx" />

Set `allowTextSelection` to `true` on the Table. This wraps cell content in a focusable container that allows text selection while still supporting row selection via checkboxes. Users can triple-click or drag to select text within cells.

<GuidelineTiles>
  <Do>
    <Do.Description>
      Enable text selection when users frequently need to copy data like IDs,
      emails, or reference numbers.
    </Do.Description>
  </Do>
  <Do>
    <Do.Description>
      Keep selection checkboxes visible so users can still select rows for
      actions.
    </Do.Description>
  </Do>
  <Dont>
    <Dont.Description>
      Don't enable text selection by default, as it adds interaction complexity
      most tables don't need.
    </Dont.Description>
  </Dont>
</GuidelineTiles>
