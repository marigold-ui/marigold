A `<Card>` represents one item in a collection an event preview, a venue tile, a product summary that repeats next to similar siblings in a grid or list. Each card groups related content (image, title, body, actions) into a single, self-contained unit and sets it apart from the page with a raised shadow.

Cards are built from four optional slots, `Card.Preview`, `Card.Header`, `Card.Body`, and `Card.Footer`. The patterns below cover the shapes a Card actually takes in product surfaces.

## Anatomy

The `<Card>` component is a compound component made up of several sub-components that map to distinct visual areas of the card.

<CardAnatomy />

- **Card.Preview**: Optional full-bleed area at the top, typically used for images or media.
- **Card.Header**: Area for the card title and supporting labels or badges.
- **Card.Body**: Main content area for descriptive text or form elements.
- **Card.Footer**: Bottom area for actions such as buttons or metadata.

## Usage

**A Card represents one item in a set.** Reach for it when the same type of item repeats in a grid or list events, products, venues, team members and each entry is the user's gateway to its own detail view. Repetition is the core case; if the element doesn't repeat, a Card is probably the wrong tool. A Card is not a standalone container to label a single region of a page with its own heading and content (for example a "Billing" or "Organizer profile" section), use a [`<Panel>`](../../layout/panel) instead.

A Card and a [`<Table>`](../table) both display a set of records; what differs is the shape of each item. Reach for a **Table** when rows share the same fields and the user wants to scan, sort, or compare values down a column. Reach for a **Card** when each item has its own visual identity media, varied content, distinct actions and the user chooses between items by recognition, not by comparing column values.

Combine Card with [`<Tiles>`](../../layout/tiles) so the grid wraps responsively as the viewport changes.

<ComponentDemo name="card-grid" />

Cards are easy to over-reach for. When repetition isn't the shape of the content, reach for one of these alternatives instead:

| Need                                            | Reach for                                                                 |
| ----------------------------------------------- | ------------------------------------------------------------------------- |
| Tabular records with shared shape               | [`<Table>`](../table)                                                     |
| Named region of a settings or admin page        | [`<Panel>`](../../layout/panel)                                           |
| A short paragraph or a single field             | [`<Headline>`](../../content/headline) + [`<Text>`](../../content/text)   |
| A status, callout, or feedback message          | [`<SectionMessage>`](../../content/section-message)                       |
| A list of similar rows with no media or actions | [`<Stack>`](../../layout/stack) with [`<Divider>`](../../content/divider) |

The sections below cover the four common misuses to avoid.

### Don't wrap tabular data in cards

When records share the same shape and users want to scan, sort, or compare across them, a [`<Table>`](../table) reads faster and handles selection, sorting, and bulk actions out of the box. A column of cards forces the eye to jump between visual blocks instead of running down a column of values.

  <Do>
    <DoFigure>
      <DoTableMockup />

    <DoDescription>
      Use a [`<Table>`](../table) when records share the same shape and users

    </DoDescription>
  </Do>
  <Dont>
    <DontFigure>
      <DontTableMockup />
    </DontFigure>
    <DontDescription>

      the ability to scan and compare values at a glance.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Don't wrap simple content in cards

A Card groups several pieces of related content. If you only have a single paragraph, a short message, or one field, the card adds visual weight without grouping anything. Use a [`<Headline>`](../../content/headline), [`<Text>`](../../content/text), or [`<SectionMessage>`](../../content/section-message) instead.

<GuidelineTiles>
  <Do>
    <DoFigure>
      <DoTextMockup />
    </DoFigure>
    <DoDescription>
      Render simple, single-purpose content as plain text. Use a
      [`<SectionMessage>`](../../content/section-message) when the content is a
      status or callout.
    </DoDescription>
  </Do>
  <Dont>
    <DontFigure>
      <DontTextMockup />
    </DontFigure>
    <DontDescription>
      Don't wrap a single short paragraph or one field in a card. The frame
      adds visual noise without grouping anything.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Don't stack many cards on a page

Cards rely on contrast against the page to feel grouped. When you stack a long column of full-width cards, every row competes with every other row, the shadows pile up, and the page becomes visually noisy. Use a list separated by [`<Divider>`](../../content/divider) for similar, row-shaped content instead.

<GuidelineTiles>
  <Do>
    <DoFigure>
      <DoListMockup />
    </DoFigure>
    <DoDescription>
      Use a list with [`<Divider>`](../../content/divider) between rows for
      long, homogeneous content.
    </DoDescription>
  </Do>
  <Dont>
    <DontFigure>
      <DontListMockup />
    </DontFigure>
    <DontDescription>
      Don't stack a long column of full-width cards. The repeated shadows and
      borders fragment the page and harm scannability.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Don't nest cards inside cards

Cards already sit on a raised surface. Nesting a card inside another card creates double elevation that no longer reads as depth, it's just a frame inside a frame. Restructure the inner content with a layout primitive like [`<Stack>`](../../layout/stack), a [`<Divider>`](../../content/divider), or split the content into sibling sections.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Place a Card on top of the default surface so the raised shadow is
      visible.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't nest a card inside another card, place it inside a
      [`<Panel>`](../../layout/panel), or sit it on any other same-elevation
      surface. The shadow has nothing to push against.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### With or without media

`Card.Preview` is optional. Whether to include an image depends on whether the visual carries information.

**Use an image preview** when each item has its own visual identity an event poster, a venue photo, a product hero. The image is part of how users recognize and choose between cards in a grid.

<ComponentDemo name="card-elevation" />

<GuidelineTiles>
  <Do>
    <DoDescription>
      Put images in `Card.Preview` so they sit full-bleed at the top of the
      card.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't nest the card's main visual inside `Card.Body`. The padding cuts the
      image off from the card's edges and the framing falls apart.
    </DontDescription>
  </Dont>
</GuidelineTiles>

**Drop `Card.Preview`** when the content is text-driven a plan summary, a status block, or a quick stat with actions. A title and short body carry the meaning, and a generic stock image only adds noise without helping users decide.

<ComponentDemo name="card-minimal" />

### Buttons in card footers

Use small buttons (`<Button size="small" />`) for actions in `Card.Footer`. Cards usually appear in a grid or list, so the footer is a tight space small buttons match the scale of the surrounding content and keep multiple cards visually balanced. Reserve default-size buttons for primary page actions outside of cards.
