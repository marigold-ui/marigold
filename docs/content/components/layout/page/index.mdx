A `<Page>` is one screen of your application. It holds the screen's title and a short description, its primary action, and the sections of content beneath them. It also applies consistent spacing and a correct heading structure, so every screen in your product feels the same.

For assistive technology, `<Page>` marks up the screen's main content region, so people can skip straight to it and hear the screen announced by its title.

<Callout title="Looking for composition guidance?">
  For where a page sits in the application frame and how to structure its
  content, see the [App Shell pattern](/patterns/layout/app-shell).
</Callout>

## Usage

### Basic structure

```tsx
<Page>
  <Page.Header>
    <Title>Billing</Title>
    <Description>Manage your plan and invoices.</Description>
    <ActionButton variant="primary">Upgrade plan</ActionButton>
  </Page.Header>
  <Panel>{/* … */}</Panel>
  <Panel>{/* … */}</Panel>
</Page>
```

### Inside an App Shell

A `<Page>` usually lives inside an [`<AppShell>`](/components/layout/app-shell), the application frame that wraps it with persistent sidebar navigation and a top bar. The frame stays in place while the page scrolls. When you don't need that frame, a `<Page>` works on its own.

### Page.Header

`<Page.Header>` sits at the top of a `<Page>`. Place a [`<Title>`](/components/content/title), an optional [`<Description>`](/components/content/description), and any actions inside it, and it arranges them for you: the title and description on the left, the actions on the right. It works the same way as [`<Panel.Header>`](/components/layout/panel) and has no sub-components of its own.

The `<Title>` becomes the page heading (an `<h1>` by default), the `<Description>` the paragraph beneath it, and the page is announced to assistive technology by its title automatically.

#### Actions

Page actions use the same slot-aware action components a `<Panel.Header>` accepts: [`<ActionButton>`](/components/actions/action-button), [`<LinkButton>`](/components/actions/link-button), [`<ActionGroup>`](/components/actions/action-group), and [`<ActionMenu>`](/components/overlay/action-menu). In a Panel these recede into quiet icon buttons. A page's primary action is meant to stand out, so it sits in the header as a prominent, label-sized primary button.

Pick the element by what the action _does_, not by how it looks. Both forms render as the same primary button:

- **Acts in place** (upgrade the plan, publish, save): use [`<ActionButton variant="primary">`](/components/actions/action-button), which renders a `<button>`.
- **Navigates** (create a new entity that opens its own page, or move to another screen): use [`<LinkButton variant="primary">`](/components/actions/link-button), which renders an `<a>`. It looks identical to the button, but because it is a real link it behaves like one (open in a new tab, copy the address, follow with the keyboard).

```tsx
// Acts in place: a real <button>
<Page.Header>
  <Title>Billing</Title>
  <ActionButton variant="primary">Upgrade plan</ActionButton>
</Page.Header>

// Navigates to a "create" page: a real <a>, styled the same
<Page.Header>
  <Title>Events</Title>
  <LinkButton variant="primary" href="/events/new">
    Create event
  </LinkButton>
</Page.Header>
```

A page has one prominent primary action, and it belongs to the page. Put it in the header for an action that operates on the whole screen, such as creating a new entity. For a form, the primary action is instead the submit button at the end of the form. A `<Panel>` is a surface within the page, so its header carries actions scoped to that surface rather than the page's primary call to action.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Make the page's primary action a `variant="primary"` action in the header.
      Use an `<ActionButton>` when it acts in place, or a `<LinkButton>` when it
      navigates (for example a "Create" action that opens a new page). Both read
      as the same primary button.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't drop a bare `<Button>` into `<Page.Header>`. It isn't slot-aware and
      won't pick up the header's grid placement. Use the slot-aware action
      components above.
    </DontDescription>
  </Dont>
</GuidelineTiles>

#### Status and metadata

`<Page.Header>` has three placements only: the title, the description, and the actions. It is deliberately not a place for object status (a `<Badge>`) or at-a-glance metadata. Show those in the page body instead, typically a secondary `<Panel>` (e.g. an "Overview" panel beside the primary content). The [detail layout](/patterns/layout/app-shell#common-page-layouts) in the App Shell pattern demonstrates this primary/secondary split.

### Heading outline

`<Page>` sets the base heading level to `1`, so a [`<Title>`](/components/content/title) inside `<Page.Header>` renders as the page `<h1>`. Each `<Panel>` below defaults to `headingLevel={2}`, so its `<Title>` renders as an `<h2>`. The outline (`h1` → `h2` → `h3` for Panel collapsibles) falls out of the defaults with no per-component configuration.

### Padding and rhythm

`<Page>` applies padding (`square-relaxed` by default) and a vertical gap between its children (`group` by default). Override padding with `p`, or `px` / `py` for per-axis control (setting `p` together with `px`/`py` is a type error). Override the gap with `space`.

### Width

`<Page>` does not impose a max-width. Content fills the `main` area, and width is decided **per surface** rather than per page. Wrap form-style content in [`<Panel size="form">`](/components/layout/panel) to constrain it to a comfortable reading measure, and leave data-dense surfaces (tables, dashboards) full-width.

```tsx
// A settings page: each surface constrained to a readable measure
<Page>
  <Page.Header>
    <Title>Account preferences</Title>
  </Page.Header>
  <Panel size="form">{/* … */}</Panel>
  <Panel size="form">{/* … */}</Panel>
</Page>
```

This is a deliberate departure from systems that expose a page-level `fullWidth` / `narrowWidth` switch. A real page often mixes a narrow form with a wide table, and constraining per surface keeps each readable without a single page-level width fighting the exceptions.

### Not sticky

`<Page>` does not stick or collapse on scroll. The document scrolls and the page header scrolls away with the content. Only the [`<AppShell>`](/components/layout/app-shell) frame (sidebar and top navigation) stays pinned. This keeps native browser scroll behaviour intact and avoids the complexity of a collapse-on-scroll header.

### Page.Content

Put `<Panel>`s directly inside `<Page>` for the common case. Reach for the optional `<Page.Content>` wrapper only when the rhythm _between_ sections should differ from the header-to-content gap. `<Page>`'s `space` then controls the header-to-content gap, and `<Page.Content space>` the gap between sections.

```tsx
<Page space="section">
  <Page.Header>…</Page.Header>
  <Page.Content space="regular">
    <Panel>…</Panel>
    <Panel>…</Panel>
  </Page.Content>
</Page>
```

### Accessibility

- **Landmark:** `<Page>` renders a `<main>` landmark. When a `<Title>` is present in `<Page.Header>`, the `<main>` is labelled by it via `aria-labelledby`, so screen-reader landmark navigation announces the page by its title.
- **No title:** If a page has no `<Title>`, pass an `aria-label` to `<Page>` so the `<main>` still has an accessible name.
