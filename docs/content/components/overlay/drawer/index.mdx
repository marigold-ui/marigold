The `<Drawer>` is a slide-in panel that hosts a supplementary, in-context task alongside the main view, without blocking it. It is non-modal on desktop, falls back to a modal sheet on small screens, and is intended for light, single-step work.

## Anatomy

A `<Drawer>` slides in alongside the main page content. It is composed of an optional title, an optional close button, the main content area, and a row of actions.

<DrawerAnatomy />

- **Page content:** The main view that stays visible and interactive while the
  drawer is open (non-modal on desktop).
- **Drawer:** The slide-in panel itself, anchored to the side of the screen and
  hosting the supplementary task.
- **Title:** Optional heading at the top of the panel that gives context for
  the task inside.
- **Close button:** Optional dismiss control in the top corner, opt in via the
  `closeButton` prop.
- **Content:** The body of the drawer, holding the main message or
  interactive elements like a form.
- **Actions:** Bottom row of buttons for confirming or cancelling the task.

## Usage

A drawer hosts a supplementary, in-context task that benefits from staying alongside the main view. Three constraints define it:

- It stays non-modal on desktop, so the page behind remains interactive.
- Its content is _about_ what is already on screen.
- It caps task complexity at roughly five fields and a single step.

When in doubt, ask: is this content _about_ the visible main view, or _for_ it? _About_ means a drawer fits. _For_ means it is a [dialog](/components/overlay/dialog) moment, or it belongs inline.

### Drawer vs Dialog

A dialog is modal: it blocks the page and asks the user to commit, acknowledge, or decide before continuing. A drawer is non-modal, so the page behind stays interactive while the user inspects, refines, or augments.

Choose a [dialog](/components/overlay/dialog) when the user must acknowledge or confirm before moving on, especially for destructive actions. Choose a drawer when the work is supplementary to whatever is already on screen.

The most common mistake here: a user is filling out a form (say, creating an event) and needs a value that doesn't exist yet, like a new category. It's tempting to open a drawer for "Create category". Don't. The new category is _for_ the form they're filling out, not _about_ the page. That's a dialog.

### Routing to a page instead

A drawer slides over the current view; a page replaces it or routes somewhere new. Drawers are ephemeral: opened, used, dismissed. Pages are durable: they have a URL, browser history, and a place in the app's information architecture. When the work needs any of that durability, it belongs on a page.

Reach for a page over a drawer when:

- the user needs a URL to share, bookmark, or return to later,
- the form has nested editable records (line items, sub-tasks with their own forms),
- the flow has multiple steps with their own commit moments (save here, then save again there),
- the user could reasonably get lost inside it without a back button or breadcrumbs to orient them.

If you find yourself wanting tabs inside a drawer, or wanting browser back/forward to work as navigation, that is the signal: it is a page.

### Detail-row inspection

Click a row in a table or list and a drawer slides in showing the record's full details. Fields are read-only or lightly editable, and the user can move on to the next row without losing the table view. Linear, Stripe Dashboard, GitHub Issues, and Salesforce Lightning record peeks all use this pattern in their list views.

The user's task here is "scan rows, drill into one." Keeping the list visible preserves orientation; routing away breaks the rhythm. If the detail view is multi-tabbed, has its own sub-records, or is something users link to directly, route to a page instead.

### Quick edit or create from a list

Users often need to make a small change to a list, like editing one row or adding a new one, without losing their place. A drawer alongside the list handles both: a small form slides in, the user saves, the drawer closes, and the list reflects the change.

The form should be small: a handful of fields, a single save, no multi-step flow. The list stays visible behind the drawer, so the surrounding rows, filters, and sort order tell the user _why_ they are working on this one.

The two flows share the same shape but differ in trigger and result:

|                              | Quick edit                                             | Quick create                                   |
| ---------------------------- | ------------------------------------------------------ | ---------------------------------------------- |
| **Trigger**                  | "Edit" on a row                                        | "Add" above the list                           |
| **Result**                   | The row updates in place                               | A new row is added to the list                 |
| **Outgrows the drawer when** | The edit needs nested records or multi-step validation | Setup is multi-step or has its own sub-records |

For the related case of creating an object that belongs to a _different_ form on the page (e.g. a category needed inside an "Edit event" form), see [Drawer vs Dialog](#drawer-vs-dialog).

### Filter panel

A collection of filter controls displayed as a side panel. The user adjusts, applies, and results update behind the panel. Non-modality is the whole point: users iterate filters and want to see the table change.

See the [filter pattern](/patterns/user-input/filter) for end-to-end guidance, including applied-filter display.

### Comments, activity, and history

The user's task here is "what's happened to this, and what should I add to it?" Long-lived records like tickets, pull requests, and issues accumulate context: comments, status changes, assignments, audit entries. A drawer slides in alongside the record and holds the whole feed, so users can scan the history, catch up on a thread, or post a reply without leaving the main view.

The pattern fits a drawer because:

- the feed grows long and needs its own scroll,
- users want the main record visible while reading or replying,
- activity is something to glance at occasionally, not the focus of the page.

If activity _is_ the primary task (a chat-first product, for example), it belongs in the main column, not in a drawer.

### Contextual utility

Persistent or summon-able utilities that support the main task without being part of it: in-app help, support chat, AI assistant, scratchpad. These utilities live alongside the main work, available when needed but never the focus.

The right shape is a drawer the user can pin or summon, not a modal or a popover. It opens when needed and never takes focus from the main task.

### Multi-field bulk edit

Users select rows in a table, an [ActionBar](/components/actions/actionbar) appears, and one of its actions opens a drawer with a small form for editing several fields across the selection (for example, changing the date and venue across many events at once). Stripe, Polaris, Airtable, and Notion all use this combination.

The drawer keeps the selected rows visible behind it (non-modal) so users can confirm the selection while editing. The boundary is clean: single-verb bulk actions (delete, archive, assign) belong in the action bar; multi-field bulk edit belongs in a drawer triggered from it. Destructive bulk actions still need a confirm dialog after the verb.

### Accessibility

A drawer renders with the ARIA role `complementary` by default, which fits almost every use case in this guide. Two exceptions are worth knowing:

- For a filter panel, consider `role="search"` to give screen-reader users a more precise landmark.
- For long-lived activity panels, consider `role="region"` with an `aria-label` if the panel is part of the persistent layout.

On small screens, the drawer falls back to a modal sheet. Focus trapping and dismissal on <kbd>Escape</kbd> are handled automatically. No extra work is required from consumers.

## Alternative components

- [Dialog](/components/overlay/dialog): Use for modal, attention-required moments like confirmations, destructive actions, and parent-form quick-create.
- [ActionBar](/components/actions/actionbar): Use for single-verb bulk actions on a selection (delete, archive, assign). Pair with a drawer when bulk-editing several fields.
- [Sidebar](/components/navigation/sidebar): Use for app-level navigation, not for in-context content alongside a record.
