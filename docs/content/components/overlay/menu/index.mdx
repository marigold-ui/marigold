The `<Menu>` component presents a list of commands behind a trigger. Reach for it when you have actions a user can take, such as editing, sharing, or deleting, and you want to keep those options tucked away until they are needed.

A `<Menu>` owns its trigger button and opens a floating list of `<Menu.Item>`s. Group related items with a `<Menu.Section>` or a `<Divider>`, and use the [`<ActionMenu>`](/components/overlay/action-menu) companion when you want a compact kebab-style trigger for per-row or per-item actions.

On small screens a `<Menu>` opens as a bottom [`<Tray>`](/components/overlay/drawer) instead of a floating list, so it stays comfortable to reach on touch.

import { MenuAnatomy } from './menu-anatomy';

## Anatomy

<MenuAnatomy />

- **Trigger:** The button that opens the menu.
- **Section:** An optional group of related items with a header that names the group.
- **Menu item:** A single command. It can carry a leading icon, a label, and a trailing keyboard-shortcut hint.
- **Shortcut:** An optional `<Keyboard>` hint that shows the item's keyboard shortcut.
- **Divider:** An optional `<Divider>` that separates groups without a label.
- **Selected item:** In selection mode, the chosen item shows a checkmark and a highlighted row.

The Menu component supports variants for both the trigger button and individual menu items.

**Trigger button variants:**

| Variant   | Description                                                        | When to use                                                                          |
| --------- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |
| `default` | Trigger button with a visible border. This is the default variant. | Most menu triggers where the button should be clearly identifiable.                  |
| `ghost`   | Transparent trigger button without a border or background.         | Menus in toolbars, table rows, or compact layouts where the trigger should blend in. |

**Menu item variants:**

| Variant       | Description                                            | When to use                                                    |
| ------------- | ------------------------------------------------------ | -------------------------------------------------------------- |
| `default`     | Standard menu item. This is the default variant.       | Regular menu actions like navigation, selection, or settings.  |
| `destructive` | Red-colored menu item signaling a dangerous action.    | Irreversible or high-impact actions like "Delete" or "Remove". |
| `master`      | Menu item with a lock icon in the master access color. | Actions reserved for Reservix employees (master access).       |
| `admin`       | Menu item with a key icon in the admin access color.   | Actions reserved for developers (admin access).                |

The item variants apply to `<Menu.Item>`, not to the trigger. See the [Admin- and Mastermark pattern](/patterns/layout/admin-master-mark) for guidance on access marks.

## Usage

A menu keeps secondary or advanced actions out of the way until a user asks for them, which keeps the surrounding interface calm. Every item is a single command: the user picks one, it runs, and the menu closes.

Order items by how often they are used, put the most common ones first, and keep labels short and free of filler verbs.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Keep item labels short and concrete, ideally under 24 characters, and lead
      with the action.
    </DoDescription>
  </Do>
  <Do>
    <DoDescription>
      Order items by priority and put the most used actions at the top.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't overload one menu with a long list of items. Group them into
      sections, or split the actions across the interface.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Menu vs Select

A menu and a [`<Select>`](/components/form/select) can look alike, but they do different jobs. A menu is a list of commands that act immediately, so choosing "Duplicate" duplicates something right away. A select is a form field that holds a value, so choosing "Germany" sets a country you submit later.

Ask whether the choice does something now or stores a value for later. If it acts, use a menu. If it fills in a field, use a select.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use a menu for actions that run as soon as they are chosen.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't use a menu to collect a value for a form. Use a
      [Select](/components/form/select) so the choice is submitted with the
      form.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Commands and actions

The most common menu is a short list of actions that each run on selection. Give the `<Menu>` an `onAction` handler and each `<Menu.Item>` an `id`, so you can tell which command was chosen.

### Grouping related items

When a menu holds several kinds of action, group them so the list stays scannable. Use a `<Menu.Section>` with a `title` when each group benefits from a visible heading, such as "Game day" and "Support". Use a `<Divider>` when the grouping is obvious and a heading would only add noise, such as setting a destructive action apart from the rest.

### Icons and keyboard shortcuts

Leading icons help users recognise an action at a glance, and a trailing keyboard-shortcut hint teaches the faster path for repeat use. Compose a `<Menu.Item>` with an icon, a [`<TextValue>`](/components/content/text-value) label, and a [`<Keyboard>`](/components/content/keyboard) shortcut. Set `textValue` on the item so type-ahead matches the label rather than the shortcut text.

<ComponentDemo name="menu-shortcuts" />

### Adding a description line

Some actions need a word of explanation. Add a [`<Description>`](/components/content/description) next to the `<TextValue>` label for a quiet second line. The description is wired to the item's accessible description automatically, so screen readers announce the label first and the detail after.

<ComponentDemo name="menu-description" />

### Toggling options

When a menu's own rows are things to switch on and off, such as which columns a table shows, set `selectionMode` to `"single"` or `"multiple"`. Selected items render a checkmark and a highlighted row, and react-aria gives each item the matching `menuitemcheckbox` or `menuitemradio` role so the state is announced correctly. The menu stays open while the user toggles.

<ComponentDemo name="menu-selection" />

This is the one case where a menu row carries state, and it works because the row itself is the toggle. If a row instead needs its own switch, second button, or link, it is no longer a menu.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use selection mode to toggle a menu's own rows, so the row itself is the
      control.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't place a switch, a second button, or a link inside a menu item.
      Screen readers cannot reach controls nested in a command, so use a surface
      that holds arbitrary content instead.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Destructive actions

Mark an irreversible action, such as deleting a record, with the `destructive` variant so its risk reads at a glance. Always confirm the action in a [`<Dialog>`](/components/overlay/dialog) before it runs, so a mis-click cannot cause data loss.

<ComponentDemo name="menu-destructive" />

### Opening a dialog from an item

When an action needs more input or a confirmation, open a [`<Dialog>`](/components/overlay/dialog) from the item. Selecting the item closes the menu and opens the dialog, where the user completes or confirms the task.

<ComponentDemo name="menu-open-dialog" />

### Disabled items

Disable an item with `disabledKeys` when the action belongs here but is temporarily unavailable, such as "Undo" with nothing to undo. Omit the item entirely when it never applies in this context, rather than showing it disabled with no explanation. Each `<Menu.Item>` needs an `id` for `disabledKeys` to reference.

<ComponentDemo name="menu-disabled" />

### Kebab menus

For per-row or per-item actions, use the [`<ActionMenu>`](/components/overlay/action-menu) companion. It shows a compact ghost trigger, switches to a `<Tray>` on small screens, and takes the same `<Menu.Item>` API.

## Accessibility

`<Menu>` follows the ARIA menu pattern. The trigger opens the list, the whole menu is a single tab stop, and users move between items with the arrow keys, jump with type-ahead, and activate with Enter or Space. Escape closes the menu and returns focus to the trigger.

Because it is a menu, an item is a single command and cannot contain interactive children. Screen readers do not expose controls nested inside a `menuitem`, and keyboard focus never lands on them. For a panel whose rows carry their own switches, buttons, or links, use a surface built for arbitrary content instead of a `<Menu>`. Toggling a menu's own rows is the exception, handled by selection mode, where the row itself is the control (`menuitemcheckbox` or `menuitemradio`).

Give the trigger a clear name through the `label` prop, or an `aria-label` when the trigger is icon-only.
