In many applications, large or complex data sets can be difficult to navigate. Filtering helps users focus on what matters most by narrowing results to meet specific criteria or comparing a smaller subset of items. By removing irrelevant results, filtering improves efficiency and makes workflows smoother.

Filters are most valuable when a data set is too large or complex for users to easily scan in its entirety. They become especially useful when multiple parameters or categories can significantly narrow the results. Well-applied filters streamline the process of finding relevant items, reduce information overload, and enable users to complete tasks more efficiently.

## Anatomy

The filtering pattern in Marigold combines consistent layout with flexible component usage. While the overall structure remains the same, the specific components used can be adapted to the data set and the application's context. This adaptability allows the pattern to support a wide range of use cases while delivering a clear and efficient user experience.

The image below shows the main areas of the pattern, their placement within the layout, and how they work together.

- **Filter Panel:** The filter panel controls, for example checkboxes, radio buttons and sliders, are shown in a panel that slides in from the right side of the page. While it overlaps part of the content, it does not block the rest of the page, so other content remains visible and interactive. This approach allows filters to be changed without breaking the flow of work, keeping the process quick and seamless.
- **Toolbar:** The toolbar is positioned above the data set and may include elements such as a search field, sorting options, and a button to open the filter panel. Placing these controls here gives users quick access to actions that refine results without requiring them to scroll. The toolbar is a layout composition built with components like [Inline](/components/layout/inline). It is intentionally not a widget with toolbar semantics, since it is dominated by a search field where arrow keys move the text cursor rather than focus.
- **Applied Filter:** This area lists the filters currently in use, allowing quick review and adjustments. It also offers a single action to clear all filters at once. This gives the user clarity about what is being displayed, prevents confusion, and enables faster refinements.
- **Data Set:** The data set is the collection of items to which filters are applied. Items that match the filter criteria are displayed, while others are hidden. This area typically takes up most of the layout because it is the primary focus for the user. Depending on the type of data, it can appear as a [table](/components/collection/table) or as items arranged in [tiles](/components/layout/tiles) and [cards](/components/content/card).
- **Pagination:** Located below the data set, the pagination divides the filtered results into smaller, manageable pages. Placing it here keeps navigation controls close to the content they affect, provides a natural reading flow from the data to the controls, and reduces cognitive load, making it easier to browse large data sets.

## Filter Types

Filters can be presented in several formats depending on the type of data and how users are expected to interact with it. The choice of filter type should reflect the nature of the data, the available space in the interface, and the intended user workflow. Common types include:

| Type         | Description                                                                                                                                                                                               | Component                                    |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| **Single**   | Allows only one option to be chosen from a set, such as selecting a single category or status. A radio group is typically used, as it makes the one-choice rule clear and supports quick decision-making. | [Radio](/components/form/radio)              |
| **Multiple** | Enables users to select several options at once, for example multiple categories or attributes. Checkboxes clearly signal that multiple selections are possible and give instant visual confirmation.     | [Checkbox](/components/form/checkbox)        |
| **Range**    | Lets users choose values within a continuous range, making it ideal for filtering quantities, prices, or measurements. Sliders provide an intuitive way to adjust values without manual input.            | [Slider](/components/form/slider)            |
| **Min/Max**  | Provides precise control over numeric ranges by allowing exact lower and upper limits to be entered. This is useful when accuracy is important.                                                           | [NumberField](/components/form/number-field) |
| **Contains** | Filters items based on specific words or phrases entered by the user. This method is valuable for keyword searches or partial matches.                                                                    | [TextField](/components/form/text-field)     |

While list-based components such as [SelectList](/components/form/selectlist) could technically cover single and multiple selection through different selection modes, prefer the dedicated form controls listed above. Radio buttons and checkboxes communicate the selection rule visually before any interaction, integrate directly with forms, and match what users know from other applications. Reach for list-based components when a filter behaves like a searchable collection, for example hundreds of options that benefit from virtualization.

## Applying Filters

How a filter commits its changes is one of the first decisions to get right, because the rest of the pattern is built on top of it. There are two models, and a well-built filter uses **both**, each in the place it fits.

<Callout title="The default split in this pattern">

- **Quick filters in the toolbar apply instantly.** Each is a single decision, so the result updates the moment the user changes it. The search field works the same way.
- **The filter panel (the slide-in drawer) batches.** It collects changes across several parameters and applies them only when the user submits with an explicit **Apply** action.

</Callout>

This split is not arbitrary. It mirrors how a user works in each place: a quick filter is one fast choice that benefits from immediate feedback, while the panel is where several criteria are composed and committed together.

|               | Instant (live)                                                                  | Batch (apply)                                                                              |
| ------------- | ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| **Commits**   | On every change                                                                 | On an explicit Apply action                                                                |
| **Best for**  | A single decision: one toolbar quick filter, the search field, a status switch  | Composing several criteria at once: the slide-in panel / drawer                            |
| **Why**       | Immediate feedback, teaches what each option does, suits exploratory refinement | Predictable number of queries, one result update instead of many, suits goal-directed work |
| **Watch out** | A query on every interaction, and results visibly reflow on each change         | One extra click, and a user can build a zero-result combination with no early warning      |

The choice follows from the cost and shape of the decision. Apply instantly for a single, cheap, fast decision. Batch when several criteria are combined or each query is expensive. The split also tracks user intent, because instant filtering suits exploration while batch suits goal-directed work. On mobile, and for multi-select controls, an explicit apply step is what users expect, since live updates there are disorienting.

### Combining Instant and Batch in One View

Because the two models live in clearly different places, a bare control in the toolbar versus a panel with its own Apply button, using both on the same screen is a deliberate choice rather than an inconsistency. Pairing instant toolbar filters with a batched "more filters" panel is a common and well-established arrangement.

What makes it work is the affordance. The Apply button and the panel container signal "compose, then commit", while a bare toolbar control signals "this takes effect now". Keep that signal strong and the two models stay legible side by side.

<Callout type="warn" title="Keep the two models visually distinct">

The real risk is a control whose apply behavior you cannot guess by looking at it, for example a multi-select dropdown sitting in the toolbar with no apply button, next to a panel that does have one. If a control could plausibly be either, users will not know whether their change has taken effect. Make instant controls look instant, and keep batched controls inside the panel behind its Apply action.

Mixing the two models is not risk-free. The safe default is to keep apply behavior uniform within one control group. Only mix instant and batched controls when you can keep them visually distinct, so a user can always tell which model a control follows.

</Callout>

### The Apply Action

For the panel's commit control, label it clearly and tell users what they are committing to:

- **Label by behavior.** Use **Apply** for a batched panel and **Done** for a panel that filters live.
- **Show the result count on the button** where the count is available, for example "Show 47 results", so users know the outcome before they commit. This is especially valuable on mobile, where it is the clearest way to preview impact.
- **Announce the change to assistive technology.** Instant or batched, a filter change updates the data set silently for screen reader users. Render the new result count in a live region so the outcome is spoken. See [Accessibility](#accessibility) for details.

## Designing Filter Options

These guidelines cover the filters you offer and the individual options inside them: which filters to include, how to order them, and how to keep long or complex option lists usable.

### Choosing and Ordering Filters

When creating filters, ensure the chosen parameters directly match actual properties or categories in the data set, so it is obvious to the user how their selections change the displayed results. Each filter's purpose should be unambiguous, allowing users to refine results in a precise and efficient way without confusion.

Organize filters in a logical order, such as by most-used, so they are intuitive to find and apply. Use clear, descriptive labels so their purpose is immediately understood. Limit the number of filters to those that provide genuine value, keeping the interface focused, uncluttered, and easy to navigate.

### Handling Many Options

When a filter offers many possible options, showing them all at once can overwhelm users and make scanning difficult. A better approach is to display only the most relevant or frequently used options first, with a "show more" button to reveal the full list when needed. This keeps the panel compact, easier to read, and still allows full access when required.

<ComponentDemo name="filter-many-options-more" />

Another helpful method is adding a filter filter, which is a small search field within the filter panel. This allows users to quickly locate specific options without scrolling through long lists. Using these techniques together makes it faster and easier to find the right criteria, reducing effort and improving the overall experience.

Place a [SearchField](/components/form/search-field) above the option list and filter the rendered options as the user types. Keep the group controlled so selections persist while their options are hidden by the search.

<ComponentDemo name="filter-options-search" />

For very long option lists in a compact space, [TagField](/components/form/tag-field) provides this behavior out of the box: it combines a search input with a virtualized option list and shows selections as removable tags.

### Showing Option Counts

Displaying counts next to filter options gives users a sense of scale and helps them make informed choices. A category labeled "Concerts (85)" or "Workshops (12)" communicates availability without the user needing to test the filter.

This is particularly helpful in large datasets where the number of matching results can guide prioritization. Counts should update dynamically as filters are applied, so that users can see immediately how their selections affect the result set. Keeping the counts visible ensures transparency and avoids the frustration of selecting filters that return little or no data.

Counts are also useful in single select filters such as those built with radio buttons. Even though only one option can be active at a time, the numbers preview the impact before a choice is made. They also reinforce clarity when a filter is already applied, showing exactly what scope remains.

Keep the count visually quiet so it informs without competing with the option itself. Use a muted [Text](/components/content/text) inside the option's label rather than a heavier element like a Badge, which draws too much attention for a secondary number. Keep it the same size as the label so the two share a baseline and stay aligned. The muted color alone is enough to make the count recede. Always place the count inside the label, regardless of the control used. This keeps the placement consistent across filter types and ensures assistive technology announces the count together with the option.

<ComponentDemo name="filter-option-counts" />

Whether single select or multi select, pairing counts inside the filter panel with a clear applied filter summary gives users confidence that their selections match their intent.

### Select All

In multi-select filters, leaving everything unchecked typically means all results are shown, so a "Select all" option is not always required. It becomes valuable in long lists where users often want nearly everything, or when a parent-child structure enables bulk toggling. In these scenarios it reduces effort and makes the common "all except a few" pattern easier to achieve.

When included, "Select all" should appear above the filter options and be paired with a quick way to reset. Partial selections are communicated with an indeterminate parent state, which assistive technology announces as "mixed". Display how many items the action affects, and use accessible labels that clarify scope, such as "Select all 125 items."

Build this with a standalone [Checkbox](/components/form/checkbox) that controls the group: it is checked when every option is selected, unchecked when none are, and indeterminate in between. Because the control sits outside the group rather than being one of its values, point its `aria-controls` at the group so the relationship is conveyed programmatically and not left to visual proximity. This follows the ARIA [mixed-state checkbox pattern](https://www.w3.org/WAI/ARIA/apg/patterns/checkbox/examples/checkbox-mixed/).

<ComponentDemo name="filter-select-all" />

### Excluding Values

Exclude filters let users say "show everything except these" instead of naming every value they want to keep. They are genuinely useful when the set to remove is small and the set to keep is large, but they add a second way to express intent, so reach for them only when inclusion alone is awkward. There are two ways to offer exclusion, and they suit different situations.

**Deselect from a full selection.** The lightest approach needs no extra control: start with every option selected and let users uncheck the few they want gone. This produces an "All except" state that is easy to grasp, keeps the panel free of operators, and fits the common case where users want nearly everything. It reads clearly in both the filter panel and the summary, replacing a long list of checkmarks with a precise description of what has been excluded. Use it for a single checkbox group where pre-checking everything is reasonable.

<ComponentDemo name="filter-exclude" />

**Switch the operator from include to exclude.** When a filter needs to flip polarity on the same set of values, give it an explicit operator such as "Is any of" / "Is none of". The selected values stay put, and only their meaning changes, so users can move a filter from include to exclude without re-picking. This model scales better than deselection when the list of values to keep would be long but the list to remove is short. Prefer it inside an advanced or per-field filter builder rather than a simple panel, and offer only one of the two mechanisms per filter so there are not two ways to express the same intent.

<ComponentDemo name="filter-exclude-toggle" />

Whichever mechanism you choose, make the exclusion unmistakable wherever the filter is shown. Phrase the applied filter as "Type is not Bar" or "All types except Bar", never as a bare value, so the polarity is never ambiguous in the panel, in the tag, or in the control's accessible name. Switching polarity can change the result set dramatically, so apply it immediately and keep the operator visible rather than buried in an advanced menu.

## Scaling to Many Filters

The techniques above help within a single filter, but a filter panel can also grow in the number of filters it contains. A panel with 15 or more filter groups forces users to scroll past many controls that are irrelevant to their current task, making the relevant ones harder to find.

### Grouping into Sections

Group related parameters into collapsible sections using an [Accordion](/components/navigation/accordion) with `allowsMultipleExpanded`. Expand the most-used groups by default and collapse the rest, so the panel opens with a compact overview while every filter stays one click away.

<ComponentDemo name="filter-group-sections" />

Be aware that collapsing comes at a cost for expert users, because it increases interaction cost when they need most of the content. If your users work with the majority of filter groups in every session, keep more groups expanded by default and offer an "Expand all" and "Collapse all" control so they can reveal or hide everything in a single action. The reliable pairing is small accordion sections for each parameter group, plus an expand and collapse all control to manage them at once.

There is also an upper limit to what a slide-in panel can hold. The panel works well because the data set stays visible while filtering, but once the number of filter groups grows far beyond what fits on one or two screens of panel height, consider a dedicated filter page or full-screen dialog instead. More filters is not automatically better: revisit whether each filter earns its place before adding structure around them.

### Quick Filters

When usage is concentrated on a few filters, hiding them in the panel adds an unnecessary step to the most common task. Establish a two-tier hierarchy: surface the few high-value filters as quick filters directly in the toolbar, and keep the full set behind a "More filters" button that opens the panel. This pairs a small set of always-visible controls for the frequent cases with a complete panel for everything else.

Promote at most 2 or 3 quick filters, chosen by actual usage, and prefer compact controls such as a [Select](/components/form/select). Quick filters apply instantly, since each represents a single quick decision, while the panel remains the place for combining several criteria and applying them together. See [Applying Filters](#applying-filters) for how the two models work side by side.

<ComponentDemo name="filter-promoted" />

<Callout title="When to promote a filter to a quick filter">

- It is among the **2 or 3 most-used** filters, based on real usage data.
- A single selection in it meaningfully narrows the data set on its own.
- It still appears **inside the panel too**, so users who open "More filters" first can still find it.

Avoid promoting a filter you cannot justify by usage. If you cannot identify the most-used few, leave everything in the panel and rely on search.

</Callout>

#### Labeling Quick Filters

How a quick filter shows its state depends on whether it allows one value or several.

For a **single select** quick filter, showing the chosen value in the trigger is fine, since only one value is ever possible. The applied filter tag for it is then optional, as the control already conveys the selection. The promoted [Select](/components/form/select) above works this way.

For a **multi select** quick filter, do not list every value in the trigger. Show the dimension label with a count instead, such as "Status (2)", and let the applied filter tags carry the actual values. This keeps the trigger compact and avoids showing the same values in two places. Label the trigger by dimension and surface the chosen values separately.

Build this with a [Select](/components/form/select) in `selectionMode="multiple"` and a `renderValue` that returns the label and count. Drive the options with an `items` collection so the selected values reach `renderValue`. When nothing is selected, the placeholder shows the dimension label on its own.

<ComponentDemo name="filter-quick-multiselect" />

#### Segmented Quick Filters

For a small set of mutually exclusive options, such as a status or a view switch, a segmented control reads as a single, always-visible choice and applies instantly. A dedicated Segmented Control component is planned ([DST-765](https://reservix.atlassian.net/browse/DST-765)). Until it lands, keep these options in the panel or use a single select [Select](/components/form/select) in the toolbar. Do not reach for `ToggleButtonGroup` here, as it is a group of independent action buttons, not a filter control.

#### Date Range Presets

Date is one of the most common things to filter by, and relative ranges such as "Last 7 days", "Last 30 days", or "This month" cover the majority of cases. Offer these as a quick filter so the frequent ranges are one click away, without making users open a calendar and count days.

Use a [Select](/components/form/select) of relative presets, applied instantly. For the cases a preset cannot express, include a "Custom range…" option that reveals an absolute range built from two [DatePicker](/components/form/datepicker) fields. Keep the Select as the single source of truth: choosing a preset clears any custom range, and entering a custom range marks the selection as "Custom".

<ComponentDemo name="filter-date-presets" />

Relative date-range presets are composed in userland today. First-class support for preset shortcuts in the date picker popover is planned ([DST-1511](https://reservix.atlassian.net/browse/DST-1511)).

## Communicating Filter State

Once filters are applied, the interface needs to reflect that state back to the user: how each active filter reads, how to signal filtering at a glance, how to undo it, and what to show when nothing matches.

### Applied Filter Labels

The recommended approach for applied filter labels is to keep them clear and structured. Combine three parts: the _identifier_ (what is filtered, such as category, property, or type), the relative or absolute _operator_ (how it applies, for example is, is above, is between, or a direct match), and the _value_ (the number, threshold, or date).

For absolute values, pair the filter name with its exact value, for example "Status is Approved", to be concise yet precise. For relative values, add the relative term and exact dates or details, such as "Last 7 days (Aug 3 to Aug 9)", to avoid ambiguity. Use clear operators like "is", "is between", "before", "after", and "in the last" for readability. Symbols (>, =, :) can work in compact or technical views but may be less clear for broad audiences, especially with relative timeframes.

Always group all selected values of one filter into a single tag, never one tag per value. "Type is Club or Lounge" reads as one decision and is removed as one, while separate "Club" and "Lounge" tags multiply quickly and obscure which filter they belong to.

Keep the identifier in the tag when a value alone would be ambiguous, for example a "Draft" status and a "Draft" tag. The choice is between an explicit label ("Status: Active") and an implied one ("Active"). Lean explicit as the number of filters grows.

When an applied filter contains a large number of values, avoid grouping them all into a single tag, as this makes scanning and reading difficult and disrupts consistent sizing. Instead, display only the first two or three values, then indicate the number of remaining ones with a clear count, for example "(+7 more)".

<ComponentDemo name="filter-applied-filter-label" />

Even with one tag per filter, many active filters can stack into several rows of tags and push the data set out of view. In that case, truncate the applied filter area after one or two rows behind a "show all" control. Built-in support for collapsing tags in `Tag.Group` is planned ([DST-1509](https://reservix.atlassian.net/browse/DST-1509)).

Marigold deliberately does not provide a helper to generate these labels. Label phrasing and translation depend on the application's domain and i18n setup, so they are best composed where the filter state lives. We will revisit a shared helper if demand shows up.

### Active Filter Count

When the filter panel is closed, users need a reliable signal that the data set is currently filtered. The applied filter area provides the details, but it can be scrolled out of view or empty space may be reclaimed by other content.

Add a [Badge](/components/content/badge) with the number of active filters to the filter button. The count is always visible alongside the trigger, signals filtering at a glance, and combined with the applied filter tags gives a complete snapshot of what is narrowing the data set.

<ComponentDemo name="filter-trigger-badge" />

### Reverting Filters

When applied filters result in no displayed results, for example when criteria are too restrictive, conflicting, or simply no longer relevant, it is important to offer quick and accessible ways to revert them. This prevents users from becoming stuck or frustrated and allows them to continue exploring the data set with minimal effort.

Two recommendations are:

1. Allow the user to remove individual filters directly from the applied filters area, enabling them to quickly discard any selection that is no longer relevant and immediately see the effect on the data set without disrupting other active filters.
2. Clear all filters at once using an action in the applied filters area. This instantly resets the filtering state and returns the view to its unfiltered default. It allows the user to remove every active filter in one simple step and immediately restore full visibility of all available data.

Offering both options provides flexibility. Users can make precise adjustments by removing specific filters or start fresh by clearing them all.

<ComponentDemo name="filter-revert" />

<Callout title='"Remove all" Visibility'>
  The "Remove all" action is automatically hidden when only one filter is
  active, as removing it individually is simpler and cleaner.
</Callout>

### Empty States

Empty states occur when no items match the applied filters, often because the criteria are too narrow, the filters conflict with each other, or the data set lacks matching values. This situation should be clearly communicated through the data set area itself, for example by displaying a clear message or visual placeholder where results would normally appear. Doing so helps users quickly understand why no results are shown and guides them toward resolving the issue. Supportive actions can include suggesting broader parameters, offering related content, or providing an easy-to-access "clear filters" control to reset the view and start fresh.

<ComponentDemo name="filter-empty-state-table" />

An empty state can also occur in the applied filters area when no filters are currently active. Displaying this state helps users understand that no filtering criteria are in place and that all available results are being shown.

## Accessibility

Filtering combines several interactive controls, so a few accessibility details make the difference between a pattern that works with a keyboard and screen reader and one that quietly excludes people.

- **Do not turn the toolbar into a `toolbar` widget.** The [WAI-ARIA toolbar pattern](https://www.w3.org/WAI/ARIA/apg/patterns/toolbar/) makes the group a single tab stop with arrow-key navigation between controls. That conflicts with a search field, where arrow keys must move the text cursor. Compose the toolbar with [Inline](/components/layout/inline) so each control is a normal tab stop, as described in the [anatomy](#anatomy).
- **Name a label-only trigger by its full state.** When a quick filter shows only the dimension and a count, make sure its accessible name still conveys that something is applied, for example "Status, 2 selected", rather than a bare "Status". With [Select](/components/form/select) the count rendered in the trigger is part of its accessible name. For a custom trigger, set an explicit `aria-label`.
- **Let the controls own focus and keyboard behaviour.** Building quick filters from Marigold components such as [Select](/components/form/select), [Checkbox](/components/form/checkbox), and [Radio](/components/form/radio) gives correct roles, keyboard interaction, and focus return to the trigger on close, because they are based on react-aria. Avoid hand-rolling `role="menu"` or a custom dropdown.
- **Announce the result of a filter change.** Applying or removing a filter changes the data set, but screen reader users get no feedback unless you announce it. Render the result count in a live region (`role="status"`, `aria-live="polite"`), for example "247 results, Status applied", so the outcome is spoken. Marigold does not yet provide a built-in region for this, so add one where the result count lives.
- **Link a "select all" control to the group it toggles.** When the control sits outside the group, as in [Select All](#select-all), set its `aria-controls` to the group's id so the relationship is conveyed programmatically and not left to visual proximity.

Some details are still application specific and not settled by the design system, such as the exact roles for a multi select filter popover. When in doubt, prefer a Marigold component that already encodes the right semantics over a custom control.

## Implementation

The reference implementation linked in the [demo](#demo) section applies a few practices that are easy to miss but make the difference between a filter that feels robust and one that surprises users. They are described here so you can apply them in your own implementation.

### Persist Filters in the URL

Applied filters, search, sorting, and pagination should always be reflected in the URL as query parameters. This means the user can bookmark a filtered view for quick access, return to the same filtered view even after closing the browser, or share it with others by sending the link. It also preserves the exact context of a search or analysis for later reference, without the need to manually reselect filters.

<Callout title="Recommendated Library">
  Use [nuqs](https://nuqs.47ng.com/) to manage URL state. This makes it easy to
  keep filter settings in sync with the address bar and share them between
  components.
</Callout>

### Reset Pagination When Filters Change

When a filter or search query changes, the result set shrinks or grows, but the current page number stays. A user on page 7 who narrows the results to 12 items would land on an empty page. Reset to the first page whenever filters or the search query change.

```tsx
const setFilter = (next: Partial<VenueFilter>) => {
  setPagination({ page: null });
  return _setFilter(next);
};
```

Wrap the state setters so the reset cannot be forgotten at individual call sites, as `useFilter` and `useSearch` do in the reference implementation.

### Keep the Filter Form in Sync

The form inside the filter panel is uncontrolled: its fields read their `defaultValue` from the current filter state once, when they mount. When a filter changes outside the panel, for example by removing an applied filter tag, the form would keep showing the stale value. Force a remount by keying the form with the filter state.

```tsx
<Drawer.Content>
  <FilterForm key={JSON.stringify(filter)} filter={filter} />
</Drawer.Content>
```

### Treat Defaults as No Filter

Some controls always carry a value even when the user does not intend to filter. A slider at its maximum or a number field at zero should mean "no filter applied". Define a sentinel value per filter and treat it as inactive: exclude it from the applied filter tags and remove it from the URL.

```tsx
const defaultFilter = {
  capacity: 0,
  price: MAX_PRICE,
  traits: [],
  rating: 0,
};

// A price at the maximum means "no price filter"
price >= MAX_PRICE ? defaultFilter.price : price;
```

This keeps the applied filter area and the URL clean, and makes "remove filter" a simple reset to the sentinel.

### Choose How Filters Enter Browser History

With filters in the URL, the browser's back and forward buttons can step through previous filter states. This is valuable for deliberate, batch-style changes, where each apply represents a meaningful state. It becomes noise for high-frequency updates such as typing into a search field, where every keystroke would pollute the history.

Use `history: 'push'` (see the nuqs [history option](https://nuqs.47ng.com/docs/options#history)) for batch-applied filter state and pagination, and keep the default `replace` behavior for continuously updating controls.
