Most business applications share a common frame: a sidebar for navigation, a top bar for context and actions, and a scrollable content area. The App Shell pattern shows how to compose this frame using [`<AppShell>`](/components/layout/app-shell), [`<Page>`](/components/layout/page), and [`<Page.Header>`](/components/layout/page-header), covering content placement decisions, the heading outline, action hierarchy, and responsive strategy.

<Callout title="Looking for the component API?">
  For props, slot details, and accessibility, see the
  [AppShell](/components/layout/app-shell), [Page](/components/layout/page), and
  [Page.Header](/components/layout/page-header) component pages.
</Callout>

## Structure

The App Shell divides the screen into three areas. The diagram below shows how they are arranged, and the code snippet shows how the components map to each area.

/>

- **Sidebar:** Sits on the left and spans the full height. It holds the application logo, primary navigation links, grouped sections, and an optional footer. Users can collapse or expand it with a toggle button.
- **Top Navigation:** Stretches across the top next to the sidebar. It should provide the sidebar toggle, breadcrumbs for orientation, and user-related controls such as account menus or status indicators.
- **Page:** Takes up the remaining space. This is the scrollable content area. Wrap it in a [`<Page>`](/components/layout/page), which renders the `main` landmark and owns the page's padding and rhythm. A page is a [`<Page.Header>`](/components/layout/page-header) followed by [`<Panel>`](/components/layout/panel)s.

[`RouterProvider`](/components/application/routerprovider) handles page navigation and `AppShell` positions everything on a CSS Grid. `AppShell` manages the sidebar's open/collapsed state internally (via `defaultSidebarOpen`), so the toggle in the header and the sidebar share state with no provider boilerplate.

```tsx
<RouterProvider navigate={navigate}>
  <AppShell defaultSidebarOpen>
    <Sidebar>
      <Sidebar.Header>...</Sidebar.Header>
      <Sidebar.Nav>...</Sidebar.Nav>
      <Sidebar.Footer>...</Sidebar.Footer>
    </Sidebar>
    <TopNavigation>
      <TopNavigation.Start>
        <Sidebar.Toggle />
      </TopNavigation.Start>
      <TopNavigation.Middle>...</TopNavigation.Middle>
      <TopNavigation.End>...</TopNavigation.End>
    </TopNavigation>
    <Page>
      <Page.Header>
        <Title>Billing</Title>
        <Description>Manage your plan and invoices.</Description>
        <ActionButton variant="primary">Upgrade plan</ActionButton>
      </Page.Header>
      <Panel>...</Panel>
      <Panel>...</Panel>
    </Page>
  </AppShell>
</RouterProvider>
```

The order of `<Sidebar>`, `<TopNavigation>`, and `<Page>` inside `AppShell` doesn't matter since the grid template handles positioning. For controlled sidebar state, render your own `<Sidebar.Provider>` around `<AppShell>` — see the [AppShell docs](/components/layout/app-shell).

### Where things go

Each area has a clear purpose. Mixing responsibilities between them makes the layout harder to use and harder to maintain.

- **Logo placement:** Put the application logo or name in the sidebar header, not in the top navigation. The sidebar is the persistent brand anchor; the top navigation is for orientation and utility actions.
- **Primary navigation:** All section-level links belong in the sidebar. Use groups (`<Sidebar.GroupLabel>`) and separators (`<Sidebar.Separator>`) to organize them. See the [Sidebar docs](/components/navigation/sidebar) for drill-down navigation and detail page patterns.
- **Top navigation:** The top navigation exists primarily for [breadcrumbs](/components/navigation/breadcrumbs) and [search](/components/form/search-field). Breadcrumbs go in the middle slot to show users where they are in the hierarchy. A search field can replace or complement them when global search is a core workflow. Don't place tabs, links, or secondary navigation here. That's what the sidebar is for. See the [TopNavigation docs](/components/navigation/topnavigation) for further information.
- **Utility actions (top navigation, right side):** The end of the top navigation is reserved for user-facing utilities like account menus, notification badges, or status indicators. Keep it concise; too many items cause the navigation to overflow on smaller screens.

### Landmarks

Each App Shell slot renders semantic HTML, so the screen reader's landmark list reflects the layout automatically.

| Slot              | Landmark        |
| ----------------- | --------------- |
| `<TopNavigation>` | `banner`        |
| `<Sidebar>`       | `complementary` |
| `<Sidebar.Nav>`   | `navigation`    |
| `<Page>`          | `main`          |

The `<main>` landmark is labelled by the page `<h1>` from `<Page.Header>`, so screen-reader landmark navigation announces the page by its title. When you compose multiple navigation regions inside the header (e.g. `<TopNavigation.Middle>` plus `<TopNavigation.End>`), give each a distinct `aria-label` so they can be told apart. Each [`<Panel>`](/components/layout/panel) inside `<Page>` adds its own `region` landmark, labelled by its title.

`F6` / `Shift + F6` cycling is opt-in via the [`useLandmark`](/components/hooks-and-utils/useLandmark) hook. Today only `<Drawer>` opts in by default — wrap the slots above (or any custom region) with `useLandmark` to add them to the cycle. See the [Landmarks foundation](/foundations/accessibility#landmarks) for details.

### Responsive behavior

`<AppShell>` always renders the L-shape grid and doesn't change at breakpoints. Responsive behavior comes from the components inside it. The sidebar collapses on desktop and becomes a modal sheet on mobile. The top navigation adapts its content at smaller widths. See the [Sidebar](/components/navigation/sidebar#collapsing) and [TopNavigation](/components/navigation/topnavigation#mobile-navigation) docs for details.

## Page structure

Inside `<Page>`, lead with a `<Page.Header>` and follow it with the page's `<Panel>`s. `<Page>` supplies the vertical rhythm between them, so you don't wrap them in a `<Stack>` or `<Inset>`.

```tsx
<Page>
  <Page.Header>
    <Title>Billing</Title>
    <Description>Manage your plan and invoices.</Description>
    <ActionButton variant="primary">Upgrade plan</ActionButton>
  </Page.Header>
  <Panel>...</Panel>
  <Panel>...</Panel>
</Page>
```

The heading outline falls out of the defaults — no per-component configuration:

| Element                            | Heading |
| ---------------------------------- | ------- |
| `<Title>` in `<Page.Header>`       | `h1`    |
| `<Title>` in `<Panel.Header>`      | `h2`    |
| `<Title>` in `<Panel.Collapsible>` | `h3`    |

`<Page>` does not enforce a max-width. For form-style pages, constrain width per surface with `<Panel size="form">` rather than wrapping the whole page.

## Actions hierarchy

Match the action to the level it belongs to. Promote a button only as high as its scope.

| Level       | Where                   | Use                                                                                           |
| ----------- | ----------------------- | --------------------------------------------------------------------------------------------- |
| **Page**    | `<Page.Header>`         | `<ActionButton variant="primary">` for the page's primary action; `<ActionMenu>` for overflow |
| **Panel**   | `<Panel.Header>`        | `<ActionButton>` (icon chrome) or `<ActionMenu>` scoped to that surface                       |
| **Row**     | inside a table/list row | `<ActionButton>` / `<ActionMenu>` scoped to the item                                          |
| **Account** | `<TopNavigation.End>`   | account menu, notifications — global utilities, not page actions                              |

A page's primary action is a prominent `<ActionButton variant="primary">` in the header; it claims the header's action slot. Don't put page actions in `<TopNavigation>` — that bar is for global utilities.

## Tabs on pages

When a page splits into tabbed views, render [`<Tabs>`](/components/navigation/tabs) as a sibling _below_ `<Page.Header>`, not inside it. The header keeps the page title and primary action; the tabs switch the content beneath.

```tsx
<Page>
  <Page.Header>
    <Title>Event settings</Title>
  </Page.Header>
  <Tabs>
    <Tabs.List>...</Tabs.List>
    <Tabs.TabPanel id="general">...</Tabs.TabPanel>
  </Tabs>
</Page>
```

## Empty state

When a page (or a Panel within it) has no data yet, render an [`<EmptyState>`](/components/feedback/empty-state) in place of the missing content. Keep the `<Page.Header>` so the page title and primary action stay available — the empty state replaces the Panels, not the header.
