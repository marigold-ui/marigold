Marigold has a growing family of containers like `<Panel>`, `<Dialog>`, `<Drawer>`, `<Tray>`, `<ContextualHelp>`, `<SectionMessage>`, and `<EmptyState>`. It also has a growing family of selection containers, including `<Menu>`, `<Select>`, `<ListBox>`, and `<ComboBox>`. They all share the same mental model: a **title** plus a **description**, often with optional **actions** alongside.

Without a shared vocabulary every container reinvents the wrappers, the props, and the styling that go with that structure. Slot configuration lets containers declare _roles_ (title, description, label, action) and lets consumers drop in shared primitives that fill those roles. The container picks the heading level, layout, and defaults. The primitives pick up the right typography automatically.

## The two patterns

Slot configuration shows up in two places. They share the same mechanism but pair different primitives.

| Surface                                                                    | Title primitive | Description primitive | Actions                           |
| -------------------------------------------------------------------------- | --------------- | --------------------- | --------------------------------- |
| **Container surfaces** (Panel, Dialog, Drawer, EmptyState, SectionMessage) | `<Title>`       | `<Description>`       | `<ActionButton>` / `<ActionMenu>` |
| **Selection items** (Menu, Select, ListBox, ComboBox)                      | `<TextValue>`   | `<Description>`       |                                   |

`<Description>` shows up in both columns on purpose. A description is the same thing whether it sits next to a Title in a Dialog or next to a TextValue in a Select option.

### Pattern 1. Container surfaces

Containers define the title region (heading level, layout) and the description region (typography), then publish their own action defaults. Consumers drop the primitives in without thinking about it:

```tsx
<Dialog>
  <Title>Delete project?</Title>
  <Description>This action cannot be undone.</Description>
  <ActionButton>Cancel</ActionButton>
  <ActionButton variant="destructive">Delete</ActionButton>
</Dialog>
```

### Pattern 2. Selection items

Selection items define the label (used to derive an item's accessible name and keyboard typeahead text) and the description (the help line below the label):

```tsx
<Select label="Plan">
  <Select.Option id="free">
    <TextValue>Free</TextValue>
    <Description>Up to 3 projects</Description>
  </Select.Option>
  <Select.Option id="pro">
    <TextValue>Pro</TextValue>
    <Description>Unlimited projects, priority support</Description>
  </Select.Option>
</Select>
```

## The rule

Use the contexts React Aria already publishes wherever they apply. Introduce Marigold-owned contexts only where React Aria has no equivalent, which today means action slots.

| Primitive              | Slot context                        | Default `slot`  |
| ---------------------- | ----------------------------------- | --------------- |
| `<Title>`              | RAC `HeadingContext`                | `'title'`       |
| `<Description>`        | RAC `TextContext`                   | `'description'` |
| `<TextValue>`          | RAC `TextContext`                   | `'label'`       |
| `<ActionButton>`       | Marigold `ActionButtonContext`      |                 |
| `<ActionButton.Group>` | Marigold `ActionButtonGroupContext` |                 |
| `<ActionMenu>`         | Marigold `ActionMenuContext`        |                 |

Container-injected styling and explicit theme styling compose cleanly. When both apply, the props get merged (rather than one overwriting the other), and Tailwind class merging keeps the output predictable.

`<TextValue>` marks itself as the option's label, which is exactly the text the option uses for its accessible name and keyboard typeahead. Screen readers announce the label as the option's name. The description is announced as supporting content.

## Building a new slot-configurable primitive

When wrapping a React Aria primitive that already has its own context (Heading, Text, Button), reuse that context. When introducing a new Marigold-owned primitive (ActionButton, ActionMenu), declare your own context.

```tsx
// 1. Declare the context (Marigold-owned only)
export const ActionButtonContext =
  createContext<ContextValue<ActionButtonProps, HTMLButtonElement>>(null);

// 2. Inside the component, consume via useContextProps
const _ActionButton = ({ ref: refProp, ...inputProps }: ActionButtonProps) => {
  const [merged, ref] = useContextProps(
    inputProps,
    refProp,
    ActionButtonContext
  );
  // Compose theme classNames with the merged context className via cn()
  // and forward the merged props + ref to the inner RAC primitive.
};

// 3. Container providers cascade defaults
<Provider values={[[ActionButtonContext, { variant: 'ghost', size: 'small' }]]}>
  {children}
</Provider>;
```

A few rules of thumb when building one:

- **Default the `slot` prop** so consumers don't have to write `slot="title"` everywhere.
- Use `useSlottedContext` (instead of `useContext`) when the context value is keyed by slot name and you want to scope the lookup to a specific slot.
- Compose `className` carefully. The merged props from `useContextProps` may carry an injected `className` from the container. Calling `cn(contextClassName, themeClassNames)` keeps both working.
- Document the precedence rules where ambiguity matters. `<ActionButton.Group>` does this explicitly. `size` cascades from the group, `variant` is overridden by the local prop (so a destructive button stays destructive inside a ghost group), and `disabled` is overridden by the local prop with the group as the fallback default.

## What's next

Phase 0 ships the primitives. Phase 1 onwards migrates the containers to consume them:

- **Phase 1.** `<Panel>` becomes the canonical real-world example.
- **Phase 2.** `<Dialog>`, `<Drawer>`, and `<Tray>` migrate.
- **Phase 3.** `<ContextualHelp>`, `<SectionMessage>`, and `<EmptyState>` migrate.

Each migration removes a layer of duplicated wrappers. The public API of those containers stays compatible while the internals collapse onto the shared primitives.
