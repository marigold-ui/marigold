The App Shell pattern provides a consistent application frame using a sidebar for primary navigation, a top navigation bar for contextual actions, and a main content area. Together, they form an L-shape layout where the sidebar spans the full height and the header stretches across the remaining width.

This pattern is suited for applications with multiple sections or modules, where persistent navigation and contextual information need to be accessible at all times.

## Anatomy

The image below shows the three main areas of the App Shell and how they are arranged.

- **Sidebar:** Persistent vertical navigation on the left side. Contains the application logo or name, primary navigation links, grouped sections, and an optional footer. The sidebar can be collapsed or expanded using a toggle.
- **Header:** Horizontal bar at the top, next to the sidebar. Holds the sidebar toggle button, breadcrumbs for orientation, and user-related controls such as account menus or status badges.
- **Main:** The primary content area, scrollable independently. This is where page-specific content is rendered, wrapped in appropriate spacing.

## Composition

The App Shell is composed of three components: [AppLayout](/components/layout/app-layout), [Sidebar](/components/navigation/sidebar), and [TopNavigation](/components/navigation/topnavigation). They are wired together using provider nesting:

1. **`RouterProvider`** - Handles navigation between pages
2. **`Sidebar.Provider`** - Manages sidebar open/collapsed state, shared between `Sidebar` and `Sidebar.Toggle`
3. **`AppLayout`** - CSS Grid container that positions the three areas

```tsx
<RouterProvider navigate={navigate}>
  <Sidebar.Provider defaultOpen>
    <AppLayout>
      <AppLayout.Sidebar>
        <Sidebar>...</Sidebar>
      </AppLayout.Sidebar>
      <AppLayout.Header>
        <TopNavigation>
          <TopNavigation.Start>
            <Sidebar.Toggle />
          </TopNavigation.Start>
          <TopNavigation.Middle>...</TopNavigation.Middle>
          <TopNavigation.End>...</TopNavigation.End>
        </TopNavigation>
      </AppLayout.Header>
      <AppLayout.Main>...</AppLayout.Main>
    </AppLayout>
  </Sidebar.Provider>
</RouterProvider>
```

## Guidelines

### Breadcrumbs for Orientation

Use breadcrumbs in the header's middle section to show the user's current location within the application hierarchy. Avoid using tabs or other navigation elements in this area.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use breadcrumbs in the header middle area to show the current page
      context.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't place tabs or additional navigation in the header middle area.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Sidebar for Primary Navigation Only

The sidebar should contain the application's main navigation structure. Avoid placing actions, forms, or content unrelated to navigation in the sidebar.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use the sidebar exclusively for primary navigation links and grouped
      sections.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't add action buttons, form fields, or unrelated content to the
      sidebar.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Responsive Header Content

The header content should adapt to available space. Keep the number of items reasonable and ensure they remain usable at different viewport sizes.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Keep header content concise and ensure it works at different widths.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't overload the header with too many elements that require horizontal
      scrolling.
    </DontDescription>
  </Dont>
</GuidelineTiles>
