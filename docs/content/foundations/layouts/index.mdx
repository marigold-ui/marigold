Most web layout is a combination of `<div>` elements with flex or grid utility classes. That works for a single container, but it scales badly. The moment a section needs a vertical stack containing a row of buttons, wrappers start nesting and each one carries its own class string:

```tsx
<div className="flex flex-col gap-4">
  <h3>Title</h3>
  <p>Body text…</p>
  <div className="flex justify-end gap-2">
    <button>Cancel</button>
    <button>Save</button>
  </div>
</div>
```

Nothing is broken here, but reading it means parsing every class string to reconstruct the intent. Repeat the pattern across a codebase and layout decisions end up scattered through utility classes, reinvented slightly differently on every page.

Marigold replaces this with named layout components. The same structure looks like:

```tsx
<Stack space="regular">
  <h3>Title</h3>
  <p>Body text…</p>
  <Inline space="small" alignX="right">
    <button>Cancel</button>
    <button>Save</button>
  </Inline>
</Stack>
```

Each wrapper states _what_ it is rather than _how_ it is configured. You can scan the tree and see the layout flow without reading a single class name. This approach has a few practical benefits:

- **Named intent.** The JSX tells you the layout pattern (`Stack`, `Inline`, `Columns`) instead of encoding it in class strings the reader has to decode.
- **One canonical pattern.** The same shape looks the same everywhere; teams don't reinvent a vertical stack in four slightly different ways.
- **Composability.** A `Button` drops into any layout without carrying layout opinions of its own.

Spacing between elements is expressed with semantic tokens like `regular` or `group`. See the [Spacing](/foundations/spacing) page for the full vocabulary.

## Components own layout, not spacing

For the benefits above to hold, components must never set their own outer margin. Adding `margin` to a component causes it to affect, or be affected by, elements outside its boundary, breaking the modularity and composition assumptions that make components reusable in the first place.

Whitespace, positioning, and arrangement are always the responsibility of a parent layout component. A button does not know what sits next to it; a `<Stack>` decides the gap between the button and its neighbour. This keeps components portable (they can be dropped into any layout without carrying layout opinions with them) and keeps whitespace predictable across pages.

## Atomic layouts

Atomic layouts are invisible positioning helpers. They do not render a surface; they exist to arrange whatever content is placed inside them. Most layouts on the web can be broken down into a handful of these patterns, combined to achieve more complex structures.

### Choosing the right layout

| Use when                                          | Component                                        |
| ------------------------------------------------- | ------------------------------------------------ |
| Stacking elements vertically with consistent gaps | [Stack](../../components/layout/stack)           |
| Laying out elements horizontally that should wrap | [Inline](../../components/layout/inline)         |
| Arranging weighted horizontal columns (e.g. 2:1)  | [Columns](../../components/layout/columns)       |
| Filling a container with equal-width tiles        | [Tiles](../../components/layout/tiles)           |
| Constraining content to a readable max width      | [Container](../../components/layout/container)   |
| Centering a child horizontally                    | [Center](../../components/layout/center)         |
| Locking a container to a fixed aspect ratio       | [Aspect](../../components/layout/aspect)         |
| Padding content inside a surface                  | [Inset](../../components/layout/inset)           |
| Building a named two-dimensional grid             | [Grid](../../components/layout/grid)             |
| Pairing a main region with a narrow sidebar       | [Aside](../../components/layout/aside)           |
| Pushing siblings to opposite ends of a row        | [Split](../../components/layout/split)           |
| Letting a region scroll independently             | [Scrollable](../../components/layout/scrollable) |
| Breaking out of a constrained grid                | [Breakout](../../components/layout/breakout)     |

## Structural layouts

Structural layouts are named, surface-bearing regions of a page. Where atomic layouts arrange arbitrary content invisibly, structural layouts define **where on the page** content lives: the frame of an application, or a labelled section within a page. They have their own visual treatment (surface, elevation, variants) and often their own slots like `Header`, `Content`, and `Footer`.

- [Panel](../../components/layout/panel) divides a page into labelled sections with a title, content, and optional actions.
- [AppLayout](../../components/layout/app-layout) is the whole-viewport frame: sidebar on the left, header on the top right, scrollable main area below.
