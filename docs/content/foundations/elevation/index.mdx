Screens are flat, but interfaces need depth. Without visual cues, every element sits on the same plane and users have to rely on position alone to understand what belongs together, what is interactive, and what is temporary. Elevation solves this by creating the illusion of layers: a card feels like it sits above the page, a dialog floats above everything else, and an input has a clear boundary against the surface it lives on.

This matters because it directly guides attention. A raised card groups related content and separates it from the surrounding page. A floating menu signals that it is temporary and will go away. A subtle border on an input tells users where they can type. Without these depth cues, interfaces feel flat and ambiguous.

Three signals work together to produce this effect:

- **Background layers** set the color of each level.
- **Boundaries** (borders) encode what kind of element something is.
- **Shadows** add visual lift to communicate how high it sits.

Boundary and shadow are deliberately independent. An element's border says whether it is a surface or a control, and its shadow says how far off the page it sits. The two compose freely and never stand in for each other.

<Callout title="Do I need to apply elevation manually?">
  Marigold components like `<Card>`, `<Dialog>`, and `<Menu>` already apply the correct background, border, and shadow tokens internally. You only need the utility classes described on this page (`ui-surface`, `ui-control`, `shadow-elevation-*`, `bg-background`, etc.) when building custom elements that aren't covered by an existing component.
</Callout>

## Background layers

Before shadows come into play, background colors establish the base layers of the interface. Three tokens define where content sits:

`bg-background` and `bg-surface` work as a pair: the page background provides contrast so that white surfaces stand out. `bg-muted` adds a gentle tint within a surface to distinguish secondary content without adding another elevation level.

## Surfaces and controls

Every elevated element in Marigold belongs to one of two families:

- **Surfaces** structure the page: cards, panels, and overlays that group, separate, and contain content.
- **Controls** are operated. Buttons, form fields, segmented controls, and calendars are all things the user manipulates to enter data or trigger an action.

Interactivity is **not** what separates them. Controls are always interactive, but a surface can be interactive too. A selectable card is still a surface. The question is the element's role. Does it contain content, or does the user operate it?

The families are told apart by their **boundary**, not their shadow:

- Surfaces wear `ui-surface`: a quiet hairline that hints at a section's edge without adding noise.
- Controls wear `ui-control`: a clearly visible boundary that makes the element stand off the page as something to operate.

```html
<!-- a surface: structure, minimal noise -->
<div class="ui-surface shadow-elevation-raised">...</div>

<!-- a control: clearly bounded, lowest shadow tier -->
<div class="ui-control shadow-elevation-border">...</div>
```

Both utilities are built on one primitive: `ui-frame` draws the shared fill, radius, and 1px rim that every raised element needs. `ui-surface` and `ui-control` are **siblings** of that frame — they differ only in which boundary token the rim wears (the quiet `surface-border` versus the denser `control-border`) and the control's subtle bottom bevel. Neither is a special case of the other: a control is not a kind of surface, it is the operable sibling.

Two consequences of the rule:

- **The border is the operability cue, and the shadow is supporting texture.** A control's border-tier shadow adds a hint of physical lift, but it never carries the affordance alone. A control stripped of its shadow, such as a selected `<ToggleButton>` or a segment inside a group, still reads as a control through its boundary and fill.
- **The boundary is required only where it is the sole cue.** A ghost button carries its affordance through its label and hover state, so it draws no border. A `<ListBox>` inside a `<Popover>` draws no boundary of its own, because the overlay frame already provides the surface. The requirement is on the system, not on every element in isolation.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Give operable elements the control boundary: `ui-control` for custom
      fields, pickers, and button-like elements.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't put `ui-control` on cards, panels, or other containers, because a
      heavy edge on a surface reads as something to click.
    </DontDescription>
  </Dont>
</GuidelineTiles>

## Shadow tiers

Elevation is split into three tiers, each serving a distinct role in the visual hierarchy. The shadow intensity increases with each tier. **Border** provides a subtle lift for controls, **raised** lifts containers off the page, and **overlay** creates a strong floating effect for temporary UI. Controls sit on the _lowest_ tier by design, because surfaces contain controls, so a card must read as the larger structure beneath the fields it holds. Together the tiers form a clear stacking order where every element sits at the right depth.

<ComponentDemo name="elevation-swatches" mode="preview" background="page" />

| Token                      | Use for                    | Components                                  |
| -------------------------- | -------------------------- | ------------------------------------------- |
| `shadow-elevation-border`  | Small interactive controls | Input, Checkbox, Radio, Switch, Tag, Button |
| `shadow-elevation-raised`  | Container surfaces         | Card, Accordion                             |
| `shadow-elevation-overlay` | Overlay elements           | Dialog, Drawer, Menu, Popover, Toast        |

This diagram shows how the three shadow tiers layer in a typical form layout. The page background (`bg-background`) is the base. A card (`shadow-elevation-raised`) groups form controls (`shadow-elevation-border`), while a floating panel (`shadow-elevation-overlay`) sits above everything.

### Border

The subtlest shadow tier. Border elevation gives controls a slight lift, defining their presence against the surface they sit on. These elements combine `ui-control` with `shadow-elevation-border`.

The edge comes from the boundary, not the shadow. `ui-control` draws the rim, and the shadow only adds a subtle lift on top. See [Token Overview](/foundations/token-overview#borders) for the border tokens.

Because border elevation produces only a subtle shadow, these controls need a raised surface behind them to be clearly visible. Placing an input directly on the page background (`bg-background`) makes its edges nearly indistinguishable from the surrounding area. A Card or similar raised container provides the contrast that border-elevated controls rely on.

<ComponentDemo name="elevation-border" />

```html
<div class="ui-control shadow-elevation-border">...</div>
```

<GuidelineTiles>
  <Do>
    <DoDescription>
      Place form controls like inputs, checkboxes, and selects inside a raised
      surface such as a Card.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't place form controls directly on the page background without a raised
      container.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Raised

One step higher than border. Raised elevation creates visual lift for container surfaces that group related content. Cards and accordions use this tier to stand out from the page background while staying below floating elements. The flat page background behind them is what makes the shadow visible. Without that contrast, the lift disappears.

<ComponentDemo name="elevation-raised" background="page" />

```html
<div class="ui-surface shadow-elevation-raised">...</div>
```

<GuidelineTiles>
  <Do>
    <DoDescription>
      Place raised surfaces on the page background (`bg-background`) so the
      shadow creates clear visual lift.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't stack raised surfaces inside each other. The similar shadow levels
      make them visually indistinguishable.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Overlay

The strongest shadow tier, reserved for elements that temporarily float above the page. Dialogs, drawers, menus, popovers, and toasts use overlay elevation. Unlike the other tiers, overlays can stack on top of one another: a confirmation dialog over a menu, or a tooltip inside a drawer. They are always dismissible and often paired with a backdrop that dims the content underneath.


<GuidelineTiles>
  <Do>
    <DoDescription>
      Use overlay elevation for temporary, dismissible elements that the user
      can close or navigate away from.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't use overlay elevation for persistent page content. If it's always
      visible, use raised elevation instead.
    </DontDescription>
  </Dont>
</GuidelineTiles>
