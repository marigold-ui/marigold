The `<Sidebar>` provides persistent, app-level navigation placed on the left side of the screen. It organizes navigation items into a vertical list, supports drill-down sub-navigation for nested sections, and adapts between a desktop panel and a mobile sheet overlay.

## Anatomy

A sidebar consists of a panel alongside the main content area. The panel is organized into three zones: a sticky header, a scrollable navigation area, and a sticky footer.

## Usage

The sidebar is the central navigation of every application. It is part of the default app layout and how users move between sections. It sits on the left side of the screen and is organized into a header for branding, a scrollable navigation area, and a footer for secondary actions. On mobile, it adapts to a sheet overlay.

### Organizing navigation

Think about what your users need to get to most often and put those items at the top. People scan vertical lists from top to bottom, so the order matters.

A common mistake is to sort items alphabetically or mirror the internal system structure. Both ignore how people actually work. An "Analytics" link sorted to the top alphabetically is unhelpful if users open it once a month, while "Orders", the thing they check ten times a day, is buried further down. Order by frequency of use, not by the alphabet or your org chart.

When the list gets long, break it into groups using section headings (`<Sidebar.GroupLabel>`) and visual dividers (`<Sidebar.Separator>`). Group items by what the user is trying to do, not by how your system is structured internally. Avoid putting user-generated or unbounded content in the navigation, since lists that can grow without limit make the sidebar hard to use.

If users need to see all items at a glance, groups with headings are the right choice. If a section has its own sub-pages that users only visit occasionally, nest them behind a branch item instead. See [drill-down navigation](#drill-down-navigation) for more on nesting.

### Labels

Keep labels short and use sentence case. The first word of each label should carry the most meaning, since users tend to scan just the beginning. "Order history" is easier to scan than "History of orders." Similarly, a label like "Orders" is better than "Transaction management module."

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use sentence case and short labels (e.g. "Order history", "Account
      settings").
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't use vague labels like "Miscellaneous" or "Other". Avoid Title Case
      for navigation items.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Drill-down navigation

Some sections have sub-pages. Instead of showing everything at once, these items open a sub-panel that slides in from the right. A back button at the top of the sub-panel returns to the previous level.

The sidebar supports two levels of navigation: the root level and one level of drill-down. Deeper nesting makes navigation harder to follow and slower to use, so we limit it to two levels.

### Showing the current page

The most important job of the sidebar is telling users where they are. Mark the current page by setting `active` on the corresponding `<Sidebar.Item>`. This highlights the item and lets assistive technologies announce it as the current page.

If the current page is inside a drill-down section, the sidebar opens that section automatically so users always see their location without extra clicks.

### Header and footer

The header is a sticky area at the top of the sidebar. Use it for your brand logo — it should be visible at all times so users know which application they are in.

The footer is a sticky area at the bottom. Think of it as a utility zone for secondary actions that should not compete with the main navigation. Common candidates are a user profile or avatar (with a dropdown for account settings and logout), a help link, or app-level settings.

### Collapsing

Users can collapse or expand the sidebar with the toggle button or the keyboard shortcut <kbd>Cmd+B</kbd> / <kbd>Ctrl+B</kbd>. The sidebar remembers this preference across page loads.

### On mobile

On screens below 640px, the sidebar becomes a sheet overlay instead of a persistent panel. It opens via the toggle button and can be closed with <kbd>Escape</kbd> or the close button. On mobile the sidebar always starts closed, regardless of the desktop preference.

### Icons

The sidebar currently does not have a dedicated icon design. If your application needs icons alongside navigation labels, this should be coordinated across all applications first to ensure consistency. For now, use text-only labels.

<GuidelineTiles>
  <Do>
    <DoDescription>Use text labels for all navigation items.</DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't add icons to navigation items without coordinating with the design
      system team first.
    </DontDescription>
  </Dont>
</GuidelineTiles>

### Accessibility

The sidebar renders as a `<nav>` element so screen readers can identify it as a navigation landmark. The keyboard shortcut <kbd>Cmd+B</kbd> / <kbd>Ctrl+B</kbd> toggles it globally. When drilling into a sub-panel, focus moves to the back button, and returns to the branch trigger when navigating back. The active item is announced as the current page. On mobile, the overlay traps focus and can be dismissed with <kbd>Escape</kbd>. Built-in strings support German and English localization.
