## Anatomy

The `<Sidebar>` uses a compound component pattern. The outermost `<Sidebar.Provider>` manages state and responsive behavior. Inside, `<Sidebar>` renders a CSS Grid with three slots: header, scrollable content, and footer.

<ul>
  <li><b>Provider:</b> Wraps the sidebar and main content. Manages open/close state, mobile detection, and context for all sub-components. Use `<Sidebar.Provider>`.</li>
  <li><b>Sidebar:</b> The panel itself, rendered as a grid with header, content, and footer slots.</li>
  <li><b>Header:</b> Sticky top area for branding, logos, or workspace switchers. Use `<Sidebar.Header>`.</li>
  <li><b>Navigation:</b> Scrollable area containing nav items with drill-down panel support. Use `<Sidebar.Nav>`.</li>
  <li><b>Item:</b> A navigation entry — either a leaf link or a branch that opens a sub-panel. Use `<Sidebar.Item>`.</li>
  <li><b>Group label:</b> A non-interactive section heading to organize items into logical groups. Use `<Sidebar.GroupLabel>`.</li>
  <li><b>Separator:</b> A visual divider between groups of items. Use `<Sidebar.Separator>`.</li>
  <li><b>Footer:</b> Sticky bottom area for user profiles, settings, or secondary actions. Use `<Sidebar.Footer>`.</li>
  <li><b>Toggle:</b> A button placed outside the sidebar (in main content) to open/close it. Use `<Sidebar.Toggle>`.</li>
</ul>

## Usage

Use `<Sidebar>` when your application has **more than five navigation items** or users need to **frequently switch between sections**. Sidebars are ideal for complex apps with deep hierarchies — SaaS platforms, enterprise tools, and B2B applications.

For simpler sites with few top-level pages, prefer horizontal navigation or `<Tabs>` instead.

- Use contrasting background to distinguish from content area
  \*/}

### Structure and layout

The `<Sidebar>` is placed on the left side of the screen alongside the main content area. The `<Sidebar.Provider>` must wrap both the sidebar and the page content so that toggling and responsive behavior work correctly.

```tsx
<Sidebar.Provider>
  <Sidebar>
    <Sidebar.Header />
    <Sidebar.Nav>
      <Sidebar.Item />
    </Sidebar.Nav>
    <Sidebar.Footer />
  </Sidebar>
  <main>
    <Sidebar.Toggle />
    {/* page content */}
  </main>
</Sidebar.Provider>
```

### Labels

──────────────────────────────────────────────────────────────

Content needed:

- Labels should be short, descriptive, in sentence case (Spectrum)
- Front-load the keyword — users scan the first word of each label (NN/g:
  "left-aligned, keyword-front-loaded labels scan best")
- Don't truncate labels — wrap text instead (Spectrum)
- Consistent behavior: either all top-level items navigate+expand, or they
  only expand — never mix (Spectrum)

Do/Don't:

- Do: sentence case, succinct labels, keyword first (e.g. "Order history")
- Don't: vague labels ("Miscellaneous"), Title Case, truncated text
  \*/}

### Labels

Use **short, descriptive, keyword-front-loaded** labels in sentence case. Users scan vertical lists quickly, so the first word of each label should carry the most meaning.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use sentence case and succinct, scannable labels (e.g. "Order history",
      "Account settings").
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
      Don't use vague labels (e.g. "Miscellaneous", "Other") or Title Case for
      navigation items.
    </DontDescription>
  </Dont>

### Drill-down navigation

──────────────────────────────────────────────────────────────

Content needed:

- Explain the drill-down pattern: branch items (items with children) open a
  sub-panel that slides in from the right
- Back button appears at top of sub-panel to return to root
- Supports ONE level of nesting (Marigold constraint; Spectrum recommends
  max 3 levels, but our drill-down pattern works best with 1)
- Width stays consistent during drill-down (no layout shift)
- Active branches auto-open on mount so the user sees the relevant section
  immediately
- Branch items auto-derive their href from their first leaf descendant

Focus management during drill-down:

- Focus moves to back button when drilling in
- Focus returns to the branch trigger when navigating back
- Non-active panels should be inert (Spectrum a11y guideline)

Demo needed:

- Sidebar with a "Products" branch that drills into sub-items
  (All products, Categories, Inventory)

Callout:

- Note that branch items act as both a navigation trigger AND a drill-down
  toggle — they navigate to the first child's href while also opening the
  sub-panel
  \*/}

### Drill-down navigation

Branch items open a sub-panel that slides in from the right, revealing nested navigation. A back button returns to the root level. This pattern supports **one level of nesting** and keeps the sidebar width consistent.

Active branches auto-open on mount so users always see the relevant section.

### Active state and current page

──────────────────────────────────────────────────────────────

Content needed:

- This is the "single most important" job of navigation (NN/g)
- Set `active` prop on <Sidebar.Item> to mark the current page
- Active item gets: aria-current="page" (a11y), distinct visual treatment
  (background, font weight, left border — defined by theme)
- When an active item is inside a branch, the branch auto-opens on mount
  so the user always sees where they are
- Failing to indicate current location is "the most common mistake" (NN/g)

Demo needed:

- Sidebar with one item marked active, showing the visual highlight
- Optionally a second demo with active item inside a drill-down branch

Do/Don't:

- Do: always set active on the current page item, provide clear visual
  differentiation
- Don't: forget to set aria-current="page", use subtle/unclear active styling
  \*/}

### Active state and current page

Indicating the current location is the **most important** job of the navigation. The active item uses `aria-current="page"` for accessibility and receives a distinct visual treatment (background, font weight, or left border).

Set `active` on `<Sidebar.Item>` to mark the current page. When an item inside a branch is active, the branch auto-opens on mount.

### Collapsing and toggling

──────────────────────────────────────────────────────────────

Content needed:

- Two ways to toggle: <Sidebar.Toggle> button and Cmd+B / Ctrl+B shortcut
- Toggle button should be placed in the main content area (not inside sidebar)
- Collapsed state = offcanvas (width transitions to 0), not icon-only
  (Shadcn documents 3 modes: offcanvas, icon, none — Marigold uses offcanvas)
- State persisted across page loads via cookie: "marigold:sidebar:state"
  with 7-day expiry (Shadcn uses localStorage; we chose cookies for SSR compat)
- Cmd+B is the de facto standard (Shadcn, VS Code, Marigold)

Demo needed:

- Interactive demo where user can click toggle or use Cmd+B

Callout:

- Mention keyboard shortcut prominently — it's a power-user feature that
  significantly improves workflow
  \*/}

### Collapsing and toggling

The sidebar can be collapsed or expanded using:

- The `<Sidebar.Toggle>` button placed in the main content area
- The keyboard shortcut <kbd>Cmd+B</kbd> (Mac) / <kbd>Ctrl+B</kbd> (Windows/Linux)

The collapsed/expanded state is **persisted across page loads** via a cookie (`marigold:sidebar:state`, 7-day expiry), so the sidebar remembers the user's preference.

### Responsive behavior

──────────────────────────────────────────────────────────────

Content needed:

- Desktop (>640px): persistent panel alongside main content, smooth CSS
  transition on width when toggling, state persisted via cookie
- Mobile (<640px): renders as a modal sheet overlay (dismissable), triggered
  by <Sidebar.Toggle>, includes close button inside, overlay with backdrop
- Breakpoint detected via useSmallScreen() at 640px
- Mobile state is independent from desktop state — always starts closed
- Focus trapped within overlay on mobile (a11y requirement)
- Escape key dismisses mobile overlay

Research backing:

- Carbon: "as a header scales down, header links should collapse into a
  left-panel hamburger menu"
- MUI: temporary drawer on mobile (<1200px), permanent on desktop
- General: touch-friendly tap targets on mobile

Demo needed:

- Ideally a demo that shows both states, or a callout explaining to resize
  the browser to see the mobile behavior

Callout:

- Mobile always starts closed regardless of cookie state
- Mobile state is independent from desktop
  \*/}

### Responsive behavior

On **desktop** (>640px), the sidebar is a persistent panel alongside main content with a smooth width transition when toggling.

On **mobile** (<640px), the sidebar renders as a **modal sheet overlay** triggered by `<Sidebar.Toggle>`. It always starts closed on mobile and includes a close button. Focus is trapped within the overlay while open.

### Accessibility

──────────────────────────────────────────────────────────────

Content needed:

ARIA attributes used (table format):
| Attribute | Where | Purpose |
|-----------------------|--------------------------|--------------------------------------|
| role="navigation" | <nav> element | Landmark recognition |
| aria-label | <nav> element | Distinguish from other nav landmarks |
| aria-current="page" | Active <Sidebar.Item> | Indicates current page |
| aria-expanded | Branch items | Expand/collapse state |

Keyboard interactions (table format):
| Key | Action |
|-----------------------|---------------------------------------------|
| Cmd+B / Ctrl+B | Toggle sidebar open/close (global) |
| Enter / Space | Activate link or toggle branch |
| Escape | Close mobile overlay |

Focus management:

- Drill-in: focus moves to back button in sub-panel
- Drill-out (back): focus returns to the branch trigger that opened the panel
- Non-active panels are inert (no focus trapping in hidden panels)
- Mobile overlay: full focus trap while open

Internationalization:

- All built-in strings (back button label, toggle label) go through
  useLocalizedStringFormatter with DE/EN support

Screen reader considerations:

- <nav> landmark with accessible label
- Semantic HTML (<nav>, <a>, <button>) used throughout
- Back button has descriptive label: "Back to {section name}"
- Close button on mobile overlay has accessible label
  \*/}

### Accessibility

The sidebar implements several accessibility patterns:

- **Keyboard shortcut:** <kbd>Cmd+B</kbd> / <kbd>Ctrl+B</kbd> globally toggles the sidebar
- **Focus management:** Focus moves to the back button when drilling into a sub-panel, and returns to the branch trigger when navigating back
- **Active state:** `aria-current="page"` on the current navigation item
- **Mobile overlay:** Focus trapping and <kbd>Escape</kbd> to dismiss
- **Internationalization:** All strings support localization (DE/EN)

## Alternative components

────────────────────────────────────────────────────────────────────────────────

Content needed:

- Tabs: same hierarchy level, switching views within a page (not app-level nav)
- Accordion: collapsing content sections, not navigation
- Breadcrumbs: show location in hierarchy, complement sidebar but don't replace it
- Drawer: for temporary overlay panels with arbitrary content (not structured nav)
  \*/}

## Alternative components

<ul>
  <li>
    [Tabs](/components/navigation/tabs): Use tabs when content groups are at the
    same hierarchy level and users need to switch between views within a single
    page, not across the application.
  </li>
  <li>
    [Accordion](/components/navigation/accordion): Use accordions for collapsing
    content sections within a page, not for app-level navigation.
  </li>
  <li>
    [Breadcrumbs](/components/navigation/breadcrumbs): Use breadcrumbs to show
    the user's location within a hierarchy. Breadcrumbs complement a sidebar but
    don't replace it.
