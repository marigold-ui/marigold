The `<TopNavigation>` is a compound component that provides a consistent, accessible navigation bar at the top of an application. It uses a three-slot layout with start, middle, and end sections, giving you full control over how branding, navigation links, and user actions are arranged.

`<TopNavigation>` is ideal for application shells where the top bar needs to accommodate a logo, primary navigation, and utility actions like search or user menus. It renders a semantic `<header>` element, while the middle and end slots use `<nav>` landmarks with appropriate ARIA labels for screen reader accessibility.

## Anatomy

The `<TopNavigation>` component uses a compound component pattern with three slots arranged in a CSS grid layout.

<ul>
  <li><b>Container:</b> The outer `<header>` element that holds the three-column grid layout. Use the root `<TopNavigation>` component.</li>
  <li><b>Start:</b> The left-aligned slot, typically used for branding such as a logo or app name. Use `<TopNavigation.Start>`.</li>
  <li><b>Middle:</b> The flexible center slot, rendered as a `<nav>` element. Commonly holds primary navigation like tabs, breadcrumbs, or a search field. Use `<TopNavigation.Middle>`.</li>
  <li><b>End:</b> The right-aligned slot, also rendered as a `<nav>` element. Typically contains utility actions such as user menus, notifications, or settings. Use `<TopNavigation.End>`.</li>
</ul>

## Usage

The `<TopNavigation>` component organizes the top-level navigation of an application into three distinct sections. Place branding content in `<TopNavigation.Start>`, primary navigation in `<TopNavigation.Middle>`, and utility actions in `<TopNavigation.End>`.

### Centering content in the middle slot

Some layouts call for centered content in the navigation bar, such as a search field. Use the `alignX` prop on `<TopNavigation.Middle>` to horizontally center its content within the available space. Note that `alignX` only applies to the Middle slot — Start and End are always aligned to their respective edges.

<ComponentDemo name="topnavigation-search" />

### Keeping navigation visible while scrolling

On pages with long content, users may need to scroll far down the page and lose access to navigation. Setting the `sticky` prop keeps the navigation bar fixed at the top of the viewport so users can always navigate without scrolling back up.

### Combining with Sidebar

For applications with multiple levels of navigation, combine `<TopNavigation>` with the [Sidebar](/components/navigation/sidebar) component. The top bar handles global concerns
like branding and utility actions, while the Sidebar provides contextual sub-navigation within each section.

The `<TopNavigation.Middle>` slot adapts to your application's navigation model. Use it for **Tabs** when the top-level pages are peers the user switches between, for **Breadcrumbs** when the page hierarchy matters and users need a trail back to parent sections, or leave it **empty** when the Sidebar alone provides sufficient navigation. This
flexibility lets the `<TopNavigation>` and `<Sidebar>` work as complementary layers — the top bar orients users globally, while the Sidebar guides them locally within the current
section.

<GuidelineTiles>
  <Do>
    <DoDescription>
      Use meaningful `aria-label` props when multiple `<nav>` landmarks exist on the page. Middle and End both render `<nav>`, so distinct labels help screen reader users differentiate them.
    </DoDescription>
  </Do>
  <Dont>
    <DontDescription>
    Don't place too many items in the navigation bar. Prioritize the most important actions and group secondary ones in menus.
    </DontDescription>
  </Dont>
</GuidelineTiles>

## Mobile Navigation

Mobile navigation should prioritize simplicity. Reduce the number of visible items and use progressive disclosure to keep the interface focused. The design system provides `useSmallScreen()` (triggers below 640px) and `useResponsiveValue()` to switch between desktop and mobile patterns. Components like Select, Menu, and ComboBox already adapt automatically — switching from Popover to Tray on small screens.

On small screens, the `<TopNavigation>` should be simplified:

- **Collapse non-essential items:** Hide labels and show icon-only actions. Use `useResponsiveValue()` or `useSmallScreen()` to control visibility at each breakpoint.
- **Breadcrumbs:** Use fewer segments on small screens. The [Breadcrumbs](/components/navigation/breadcrumbs) component supports collapsing via the `maxVisibleItems` prop.

## Accessibility

The `<TopNavigation>` renders a semantic `<header>` element, while `<TopNavigation.Middle>` and `<TopNavigation.End>` each render a `<nav>` landmark. When both are present, screen readers detect multiple navigation landmarks, so provide distinct `aria-label` values to help users differentiate them (e.g., "Main navigation" vs "User actions").
